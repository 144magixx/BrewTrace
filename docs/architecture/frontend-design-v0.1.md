# 前端工作台设计 v0.1

## 1. 文档目标

本文沉淀咖啡品鉴创作 Agent MVP 的前端设计方向。当前只确认信息架构、视觉风格和关键交互，不进入代码实现。

## 2. 设计气质

已确认方向：

- 主体是专业创作工作台。
- 轻微融合 Agent 调试感。
- 咖啡氛围只通过一个简洁线条风咖啡豆 Logo 表达。
- 不做复古咖啡馆、手账、奶油色、大面积棕色或营销页风格。

关键词：

- 白色为主。
- 极低饱和度冷色调。
- 专业、克制、科技风。
- 信息密度适中，服务长期使用。

## 3. 总体布局

采用三栏工作台：

```text
左侧导航栏 | 中间主工作区 | 右侧 Agent 轨迹栏
```

### 3.1 左侧导航栏

导航项：

- 当前记录
- 历史记录
- 风味词库
- 用户偏好
- 发布记录
- 设置

设计规则：

- 固定窄栏。
- 顶部放咖啡豆线条 Logo。
- 每项使用图标 + 短文字。
- 当前选中项使用极低饱和冷色背景或左侧细线标记。
- MVP 可先完整保留入口，其中“当前记录”为核心实现页。

### 3.2 中间主工作区

默认打开“当前记录”，首屏像 Codex 一样进入对话创作：

```text
今天喝了什么咖啡？

[Agent 模式：显式工作流 | 模型自主工具调用]

[ 对话创作流 ]

[ 底部固定文本框 ]
```

中间主工作区内部再分两块：

```text
左：对话创作流
右：当前记录紧凑信息面板
```

设计意图：

- 首屏足够轻，用户可以直接说今天喝了什么。
- Agent 通过追问逐步补齐信息。
- 结构化记录在旁边自动沉淀，而不是一开始压给用户完整表单。
- 顶部提供 Agent 编排模式切换，用于对比“显式工作流”和“模型自主工具调用”两种 Agent 行为。

### 3.2.1 Agent 模式切换

中间主工作区顶部放置一个轻量 segmented control：

```text
Agent 模式  [显式工作流] [模型自主工具调用]
```

模式说明：

- 显式工作流：默认模式。Planner 输出结构化步骤，Orchestrator 逐步执行，稳定、可解释、便于学习规划链路。
- 模型自主工具调用：实验模式。基于 Spring AI Advisor / Tool Calling，让模型在受控工具集合内自主选择工具，便于学习模型如何决策调用工具。

交互规则：

- 模式切换只影响后续对话轮次，不改写已完成轨迹。
- 当前模式必须在右侧 Agent 轨迹卡片中显示。
- 切到模型自主工具调用模式时，用极轻量提示说明高影响动作仍需二次确认。
- 发布、互动、生图等高影响或产生成本动作，不因切换模式而跳过确认。

### 3.3 右侧 Agent 轨迹栏

采用卡片流，而不是终端日志或时间线列表。

每种信息类型使用一种低饱和颜色卡片：

| 类型 | 视觉建议 | 内容 |
|---|---|---|
| 真实 Prompt | 浅蓝 | 传入模型的真实 prompt 或引用 |
| 上下文组装 | 冷青灰 | 本次进入模型的事实、联想、外部参考边界 |
| 模型决策 | 浅紫灰 | 模型为什么追问、生成、拒绝或调用工具 |
| 记忆召回 | 浅绿灰 | 召回的历史记录、相似原因和使用状态 |
| 工具调用 | 浅橙 | 工具名称、目的、入参摘要、返回摘要 |
| 审稿风险 | 浅红 | 未确认事实、过度夸张、重复表达等风险 |
| 系统状态 | 浅灰 | 流程开始、结束、失败、降级等状态 |
| 编排模式 | 浅靛蓝 | 本轮使用显式工作流还是模型自主工具调用 |
| 错误与降级 | 浅红/浅黄 | 错误分类、失败原因、可恢复状态和下一步动作 |

卡片规则：

- 按时间顺序排列。
- 卡片默认展示摘要。
- 右下角显示年月日时分秒。
- 点击卡片查看详情。
- 详情可展示完整 prompt、模型输出、工具入参、工具返回、召回内容和决策理由。
- MVP 默认不做内容脱敏，但不得记录完整 API Key。
- 模型自主工具调用模式下，工具调用卡片必须展示“模型为什么选择该工具”。

数据来源：

- 首次进入工作台时，通过 REST 获取 workspace 快照。
- 用户提交对话或触发工具后，通过 SSE 订阅当前会话事件流，实时追加轨迹卡片。
- SSE 断线重连后，先重新拉取 workspace 快照，再继续订阅增量。
- SSE 只展示状态和轨迹，不承载发布、生图、互动等确认动作。

错误展示规则：

- `USER_FIXABLE`：提示用户处理方式，例如登录、验证码、补字段。
- `RETRYABLE`：显示重试入口，但不自动重复高影响动作。
- `DEGRADED`：说明已降级但流程可继续，例如无外部参考创作。
- `FATAL`：提示当前流程不可继续，并保留可恢复信息。
- `SAFETY_BLOCKED`：明确说明因缺少确认或安全策略被阻断。
- 错误卡片必须保留原始流程上下文，不遮挡或清空用户已填写内容。

## 4. 当前记录紧凑面板

右侧小面板使用专业工具属性面板风格，不做大卡片堆叠。

内容结构：

```text
当前记录
状态：信息收集中

咖啡豆
豆名      待确认
烘焙商    待确认
产区      埃塞，待确认
处理法    水洗，待确认

冲煮
滤杯      V60
水温      92°C
粉水比    待补充
分段      待补充

风味
柑橘      待细化
红茶      已确认

待确认
具体柑橘类型
文案风格
是否归档
```

交互规则：

- 信息项可点击编辑。
- 待确认项轻微高亮。
- 已确认事实、模型联想、外部参考和待确认信息必须有视觉区分。
- 面板可进入完整模板编辑，但默认保持紧凑。

## 5. 风味联想组件

采用“候选 chips + 展开说明”。

示例：

```text
柑橘可以更具体一些

[柠檬] [青柠] [甜橙] [血橙] [葡萄柚] [蜜柑] [柚子]

甜橙
圆润、亲和、甜感明显
中温 · 味道 · 甜感/酸质
[接受] [编辑] [不适合]
```

规则：

- 支持多选。
- 支持拒绝和编辑。
- 被接受的候选进入当前记录。
- 未接受的候选只能作为模型联想，不得写入事实记录或最终文案事实。
- 候选必须显示温度段、香气/味道、适用感知维度和正负倾向。

## 6. 感官维度图

当前记录面板中放迷你雷达图，点击进入完整编辑。

完整编辑使用：

- 雷达图展示整体强度轮廓。
- 0-10 滑杆精确输入。

基础维度：

- 酸质
- 甜感
- 苦感
- 醇厚度
- 干净度
- 余韵

规则：

- 文案必须表达“强度”，不表达高级或低级。
- 后续可扩展香气强度、发酵感、涩感、茶感等维度。

## 7. 文案草稿展示

采用 Tabs 切换三种文案版本：

```text
文案草稿

[克制版] [夸张版] [锐评版]

标题
正文
标签

事实边界
审稿提示

[设为最终版] [调整语气] [生成发布包]
```

规则：

- 默认打开克制版。
- 每个版本都显示标题、正文、标签、事实边界和审稿提示。
- 事实边界包括已确认事实、模型联想、外部参考和待确认信息。
- 选定最终版后，才能归档、生成发布包或发布。

## 8. 发布预览与确认

采用“发布包检查页 + 两步确认按钮”。

触发方式：

- 用户在聊天中明确表示“发布吧”“发小红书”“ok 你发吧”时弹出。
- 表单或文案草稿准备完整后，用户点击“生成发布包”时弹出。
- 信息不足时先提示缺失项，不直接弹发布确认。

流程：

```text
生成发布包
-> 用户确认发布包
-> 填写小红书发布页
-> 用户浏览器预览
-> 二次确认公开发布
-> 发布成功 / 失败 / 取消保留草稿
```

发布包检查页展示：

- 标题
- 正文
- 标签
- 图片或配图建议
- 发布前检查项
- 小红书登录状态
- 工具调用状态

规则：

- “填写小红书发布页”和“公开发布”必须是两个独立动作。
- 公开发布按钮默认禁用，只有完成发布页填写并由用户二次确认后才可用。
- 发布失败或取消时保留发布包和失败原因。
- 右侧 Agent 轨迹栏必须记录发布相关工具调用。

## 9. MVP 页面优先级

P1：

- 当前记录页。
- 对话创作流。
- Agent 模式切换。
- 当前记录紧凑面板。
- 风味联想组件。
- 文案草稿 Tabs。
- Agent 轨迹卡片栏。

P2：

- 完整模板编辑。
- 感官维度完整编辑。
- 历史记录页。
- 用户偏好页。

P3：

- 风味词库管理页。
- 发布记录页。
- 设置页中的高级工具配置。

## 10. 待实现前确认项

- UI 组件库最终是否采用 shadcn/ui + Radix UI。
- 图标库是否使用 lucide-react。
- 雷达图使用 ECharts 还是 Recharts。
- Agent 轨迹详情是内联展开，还是右侧栏内二级详情视图。
- Logo 是否由前端 CSS/SVG 绘制，还是使用独立图像资源。
