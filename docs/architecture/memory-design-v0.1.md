# 长短期记忆设计 v0.1

## 1. 文档目标

本文设计咖啡品鉴创作 Agent 的记忆系统。目标是让系统既能记住当前会话，又能长期学习用户偏好、历史咖啡记录和文案风格。

## 2. 设计原则

- 短期记忆服务当前任务，生命周期短。
- 长期记忆服务后续召回，必须结构化、可检索、可更新。
- 用户确认的信息优先级高于模型推断。
- 模型联想必须标记为候选，不直接进入事实记忆。
- 记忆召回要按需加载，避免把所有历史塞进上下文。

## 3. 记忆分层

```text
Short-term Memory
  -> 当前会话状态
  -> 已收集字段
  -> 待确认信息
  -> 当前草稿

Long-term Memory
  -> 用户偏好
  -> 咖啡记录
  -> 文案归档
  -> 风味关键词
  -> 相似记录索引
```

## 4. 短期记忆

短期记忆保存在会话中，用于多轮对话和模板填写。

核心字段：

- `sessionId`
- `currentIntent`
- `coffeeDraft`
- `brewParamsDraft`
- `sensoryScoresDraft`
- `flavorDraft`
- `confirmedFacts`
- `rejectedSuggestions`
- `pendingQuestions`
- `currentDraftTexts`

示例：

```json
{
  "currentIntent": "GENERATE_NOTE",
  "confirmedFacts": ["水洗埃塞", "高温有柑橘感"],
  "rejectedSuggestions": ["葡萄柚苦韵"],
  "pendingQuestions": ["想要克制版还是夸张版？"]
}
```

短期记忆退出条件：

- 用户明确结束会话。
- 最终文案已确认并存档。
- 会话长时间未继续，进入待恢复状态。

## 5. 长期记忆

长期记忆用于跨会话学习。

### 5.1 用户偏好记忆

记录用户长期偏好。

字段：

- 文案偏好：克制、夸张、锐评、专业、生活化。
- 口味偏好：酸、甜、苦、花香、发酵感、茶感等。
- 表达禁忌：不喜欢太油腻、不喜欢空泛、不喜欢过度种草等。
- 发布习惯：标题长度、标签偏好、是否常用封面文案。

更新规则：

- 用户显式表达偏好时直接更新。
- 多次选择同类文案后，可以生成候选偏好，但需标记为推断。
- 重要偏好变更前要让用户确认。

### 5.2 咖啡记录记忆

每次咖啡品鉴形成一条记录。

核心字段：

- 咖啡豆基础信息。
- 冲煮参数。
- 感官维度评分。
- 高温、中温、低温风味。
- 香气和味道描述。
- 用户最终选择的文案。
- 风味关键词。
- 用户满意度。
- 是否发布。

### 5.3 文案归档记忆

文案归档用于避免重复表达。

记录：

- 标题。
- 正文。
- 标签。
- 文案风格。
- 核心表达句。
- 禁止短期内重复使用的词组。

### 5.4 风味关键词索引

每条记录都需要生成索引字段。

示例：

```json
{
  "flavorKeywords": ["柑橘", "甜橙", "红茶", "蜂蜜"],
  "temperatureStages": ["高温", "中温", "低温"],
  "sensoryProfile": {
    "acidity": 7,
    "sweetness": 6,
    "body": 4
  }
}
```

这些字段用于后续召回类似咖啡。

## 6. 召回策略

一次生成文案前，Memory Service 应按优先级召回：

1. 明确同一支咖啡记录。
2. 同产区、同处理法、同风味关键词记录。
3. 感官维度相似记录。
4. 用户近期满意的同风格文案。
5. 用户明确不喜欢的表达。

召回结果需要控制数量，第一版建议每类最多 3 条。

## 7. 存储方案决策

### 7.1 结构化存储

已确认：PostgreSQL。

用途：

- 咖啡豆信息。
- 冲煮参数。
- 感官评分。
- 风味记录。
- 文案归档。
- 用户偏好。
- 会话摘要。

### 7.2 语义检索

已确认：pgvector。

Embedding 已确认使用阿里云百炼 OpenAI 兼容接口：

- 模型：`text-embedding-v4`
- 默认维度：1024
- 配置：`EMBEDDING_BASE_URL`、`EMBEDDING_API_KEY`、`EMBEDDING_MODEL`、`EMBEDDING_DIMENSIONS`

用途：

- 历史文案相似度检索。
- 风味描述相似度检索。
- 用户偏好摘要召回。
- 相似咖啡记录召回。

索引策略：

- 已确认：MVP 初期使用 HNSW 索引。
- 向量列使用 `vector(1024)`，默认 cosine 距离，索引操作符使用 `vector_cosine_ops`。
- 第一版参数使用 pgvector 常见默认值：`m = 16`、`ef_construction = 64`。
- 查询参数 `hnsw.ef_search` 初期沿用默认值，后续用集成测试和真实记录调优。
- HNSW 查询结果必须与结构化字段检索结果合并、去重，并输出“相似原因”。

示例：

```sql
CREATE INDEX memory_embeddings_vector_hnsw_idx
ON memory_embeddings
USING hnsw (vector vector_cosine_ops)
WITH (m = 16, ef_construction = 64);
```

实现建议：

1. 结构化字段检索和 pgvector 并行设计。
2. MVP 初期即保留 embedding 写入和向量召回链路。
3. 字段检索与向量检索结果需要合并、去重和排序。
4. 对比字段检索与向量检索的召回效果。

## 8. 记忆写入流程

```text
用户确认最终文案
-> 生成归档记录
-> 提取风味关键词
-> 生成文案风格标签
-> 更新用户偏好候选
-> 写入结构化数据库
-> 通过 Kafka 事件异步生成 embedding
-> 写入 memory_embeddings
```

## 9. 记忆读取流程

```text
用户发起新任务
-> 从当前上下文提取检索条件
-> 检索相似咖啡记录
-> 检索相似文案
-> 检索用户偏好
-> 压缩成可读上下文
-> 提供给文案 Agent 和审稿 Agent
```

## 10. 学习重点

学习这个模块时，重点关注：

- 短期记忆和长期记忆的边界。
- 什么信息可以直接存，什么信息必须确认后再存。
- 如何把业务对象变成可检索 metadata。
- RAG 不是简单把资料塞进 prompt，而是检索、筛选、压缩、引用。
- 记忆系统的失败通常来自召回过多、召回不准、事实和联想混淆。

## 11. 待确认问题

- 用户偏好已确认允许系统自动推断，但必须保存依据，并允许用户编辑或删除。
- 历史记录是否需要支持导入小红书旧笔记？
- 文案归档是否需要保存发布后的 URL 和互动数据？
