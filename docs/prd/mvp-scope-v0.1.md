# MVP 需求拆解与里程碑 v0.1

## 1. 文档目标

本文将 PRD v0.1 拆解为可实施的 MVP 范围和里程碑。当前版本用于对齐先做什么、暂缓什么，以及每一阶段要学会哪些 Agent 开发能力。

## 2. MVP 目标

第一版目标：

- 用户可以记录一次咖啡品鉴。
- 用户可以通过对话或模板补齐信息。
- 系统可以做风味联想。
- 系统可以生成小红书发布包。
- 系统可以归档本次记录，并在后续召回相似记录。

成功标准：

- 5 分钟内完成一次记录和文案生成。
- 至少生成克制版、夸张版、锐评版三种文案。
- 历史记录可以按风味关键词或咖啡信息召回。
- 用户能看懂一次 Agent 从输入到输出的执行链路。

## 3. MVP 功能范围

### 3.1 必做

- 咖啡豆信息手动填写。
- 冲煮参数填写。
- 感官维度 0-10 评分。
- 高温、中温、低温下的香气和味道记录。
- 风味关键词联想。
- 对话式信息补齐。
- 三种风格文案生成。
- 发布包生成。
- 本次记录归档。
- 历史记录召回。

### 3.2 应做

- 豆袋图片解析。
- 文案重复检测。
- 用户偏好记忆。
- 审稿 Agent 检查虚构和夸张表达。
- 图片生成接口预留。
- 基于 `xiaohongshu-skills` 的搜索和发布页填写。

### 3.3 暂缓

- 真实一键发布小红书。
- 完整小红书 RAG 检索和批量抓帖。
- 完整前端工作台。
- 复杂多 Agent 独立部署。
- 运营数据分析。

## 4. 里程碑

### M1：文档与技术选型确认

目标：

- 完成 PRD、架构、记忆、调研、MVP 文档。
- 确认后端框架、构建工具、数据库、向量检索、前端形态、模型接入方式。

产出：

- 当前文档集。
- [技术选型决策记录 v0.1](../architecture/technology-decisions-v0.1.md)。

学习重点：

- 如何从需求拆架构。
- 如何识别技术选型中的待确认项。

### M2：后端骨架和领域模型

目标：

- 初始化 Java 后端。
- 建立咖啡记录、冲煮参数、风味、文案草稿等领域模型。
- 提供基础 API。

学习重点：

- 领域建模。
- Java 后端项目结构。
- API 和业务服务分层。

### M3：对话模式和短期记忆

目标：

- 支持创建会话。
- 支持多轮对话补齐信息。
- 支持短期记忆保存当前上下文。

学习重点：

- Agent 上下文管理。
- 会话状态设计。
- 模型输出结构化。

### M4：模板模式和风味联想

目标：

- 支持结构化字段填写。
- 支持感官评分。
- 支持风味关键词联想和用户选择。

学习重点：

- 结构化输入如何进入 prompt。
- 词库、规则和模型联想如何配合。

### M5：长期记忆和检索召回

目标：

- 支持品鉴记录归档。
- 支持按咖啡信息、风味关键词、感官评分召回历史记录。
- 支持用户偏好记录。

学习重点：

- 长期记忆设计。
- RAG 的检索、筛选、压缩。
- 结构化检索和向量检索的取舍。

### M6：文案生成和审稿

目标：

- 生成三种风格文案。
- 审稿 Agent 检查虚构、重复、风格偏差。
- 输出发布包。

学习重点：

- 多 Agent 角色拆分。
- 输出约束。
- 文案质量评估。

### M7：多模态与外部工具

目标：

- 接入豆袋图片解析。
- 接入图片生成。
- 接入 `xiaohongshu-skills`，支持小红书搜索和发布页填写。
- 预留正式发布接口，发布动作必须二次确认。

学习重点：

- 多模态输入输出。
- 工具调用协议。
- 高风险工具的用户确认机制。

## 5. 技术选型确认结果

已确认：

- 后端：Spring Boot 4.x。
- AI 集成框架：Spring AI 2.x。
- 构建：Maven。
- 持久化：Domain 对象保持纯净；业务表用 Spring Data JPA；pgvector 向量查询用 JDBC/SQL。
- 数据库：PostgreSQL。
- 向量检索：pgvector。
- 向量索引：HNSW，默认 cosine 距离。
- 用户模型：本地单用户，固定 `local-user`，不做登录鉴权。
- 本地基础设施：Docker Compose 管理 PostgreSQL/pgvector、Kafka、Kafka UI 与 Redis。
- 前端：React + Vite + TypeScript 完整工作台。
- 模型：用户提供 `gpt-5.5` 和 `gpt-image-2`，API Key 形如 `sk-xxxxx`。
- 密钥配置：真实 Key 放在仓库外私有 env 文件，例如 `~/.config/xhs-coffee-agent/env`。
- 配置绑定：Spring Boot `@ConfigurationProperties` 按模型、Embedding、小红书、Agent、存储、Redis 等子系统分组。
- API 响应格式：REST JSON 响应统一使用 `{ requestId, data, error }` envelope。
- 模型网关：自定义 `ModelGateway` 封装 Spring AI。
- Prompt 管理：放在 `src/main/resources/prompts/`，按场景和版本管理。
- 文件存储：本地文件系统，路径入库。
- Redis：第一版仅预留配置和健康检查，不进入核心链路。
- 后端包结构：按业务边界优先，每个主要边界内分 `api/application/domain/infrastructure`。
- 外部检索和发布：默认使用本地 `xiaohongshu-skills`。

仍需确认：

- Spring Boot 4.x 与 Spring AI 2.x 的具体补丁版本。
- Java 版本，默认建议跟随 Spring Boot 4.x 要求选择 Java 21 或更高版本。
- UI 组件库最终方案。
- `xiaohongshu-skills` 的发布确认流程细节。

## 6. 验收方式

每个里程碑完成后都需要：

- 给出功能说明。
- 给出本地运行或验证步骤。
- 说明已验证和未验证内容。
- 更新相关文档。
- 如果技术选型发生变化，补充决策记录。

## 7. 第一版学习产物

除代码外，应持续沉淀：

- Agent 执行链路说明。
- Prompt 和工具调用样例。
- 记忆召回样例。
- 失败案例复盘。
- 技术选型决策记录。
- 后端设计文档。
- 技术选型学习文档，统一放在 `docs/learn/`。

目标不是只做出一个工具，而是让用户能讲清楚这个 Agent 系统为什么这样设计、如何运行、如何继续迭代。
