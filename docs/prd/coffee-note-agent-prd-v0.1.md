# 咖啡品鉴创作 Agent PRD v0.1

## 1. 文档信息

- 文档状态：草案
- 创建日期：2026-06-29
- 产品方向：面向小红书咖啡博主的咖啡品鉴记录与创作 Agent
- 当前阶段：需求梳理与 MVP 定义

## 2. 背景

用户是一名 Java 后端开发者，同时是小红书咖啡博主。用户平时会记录每次喝的新咖啡，且希望尽量不重复喝同一支咖啡。每次记录通常包含品鉴感受、冲煮参数和发布文案。

当前痛点包括：

- 每次手写品鉴记录和小红书文案较累，创作成本高。
- 遇到相似风味时容易词穷，例如总是用“柑橘”概括所有柑橘类风味。
- 缺少稳定的评价体系，导致不同咖啡之间难以对比。
- 冲煮参数、风味表现和最终文案没有形成可复用记忆。
- 希望通过真实项目熟悉 Agent 开发全链路，包括上下文管理、规划、记忆、工具加载和多 Agent 协作。

## 3. 产品目标

构建一个“咖啡品鉴创作 Agent”，帮助用户完成从咖啡记录到小红书发布文案生成的流程。

核心目标：

- 降低咖啡笔记创作成本。
- 建立一致的咖啡品鉴记录体系。
- 辅助用户扩展风味表达和文案想象力。
- 基于历史记录避免文案重复、风味描述重复和咖啡记录混乱。
- 通过真实业务场景沉淀 Agent 工程能力。

非目标：

- 第一版不追求严格复刻 SCA 或其他专业杯测体系。
- 第一版不把自动发布小红书作为强依赖能力。
- 第一版不做复杂社交媒体运营系统。
- 第一版不追求完全自动创作，关键发布动作仍需用户确认。

## 4. 目标用户

第一阶段仅面向项目作者本人。

用户特征：

- 熟悉 Java 后端开发。
- 有 Agent 开发学习诉求。
- 长期记录咖啡体验。
- 希望文案克制、真实，但也希望在需要时能生成更夸张、更有冲击力的表达。
- 对咖啡风味有一定感知，但希望系统帮助细化描述。

## 5. 核心使用场景

### 5.1 对话式创作

用户通过自然语言描述本次咖啡体验，Agent 通过追问补齐信息，并生成文案。

示例：

- 用户：“今天喝了一支埃塞，水洗，高温有点柑橘，凉下来像红茶。”
- Agent：追问豆子信息、冲煮参数、酸甜苦表现、想要文案风格。
- Agent：生成标题、正文、标签，并询问是否存档。

### 5.2 结构化记录

用户通过模板填写咖啡豆信息、冲煮参数、感官评分和风味表现。

系统在填写过程中提供：

- 豆袋图片解析。
- 风味名词自动联想。
- 风味描述辅助选择。
- 高温、中温、低温不同阶段的香气和味道记录。
- 感官维度强度图。

### 5.3 相似内容参考

用户询问“别人是什么感受”或系统判断需要参考外部内容时，Agent 检索小红书等公开内容中与当前咖啡或相似风味相关的帖子，用于辅助文案灵感。

约束：

- 外部内容仅作为参考，不直接复制。
- 需要保留来源和检索关键词。
- 需要区分用户真实感受与外部参考表达。

### 5.4 文案发布

Agent 生成发布包，包括：

- 标题
- 正文
- 标签
- 图片建议或 AI 生成图
- 发布前检查项

用户确认后，系统尝试调用发布工具。

当前结论：

- 小红书一键发布能力需要继续调研官方开放能力。
- 第一版先实现发布包生成和用户确认流程。
- 发布工具预留插件化接口，不把真实发布作为 MVP 阻塞项。

### 5.5 风味视觉化

用户可以主动要求根据本次咖啡风味生成图片。

示例：

- “这支咖啡让我想到雨后的橙子园，帮我画一张图。”
- “根据花香、青柠、红茶感做一张小红书封面。”

系统调用图像生成模型生成配图或封面素材。

## 6. 功能需求

### 6.1 咖啡豆信息管理

字段包括：

- 咖啡豆名称
- 烘焙商
- 产区
- 国家
- 庄园或处理站
- 品种
- 处理法
- 烘焙度
- 烘焙日期
- 官方风味描述
- 购买渠道
- 豆袋图片

能力要求：

- 支持手动填写。
- 支持上传豆袋图片并解析信息。
- 解析结果必须允许用户修改。
- 解析结果应标记置信度或来源，避免把识别结果伪装成事实。

### 6.2 冲煮参数记录

字段包括：

- 冲煮方式
- 滤杯或器具
- 滤纸
- 粉量
- 水量
- 粉水比
- 水温
- 研磨度
- 注水分段
- 总时长
- 搅拌、闷蒸、绕圈等手法备注
- 用水信息

能力要求：

- 支持模板化填写。
- 支持常用参数默认值。
- 支持后续复盘时按参数检索。

### 6.3 感官维度图

感官信息采用 0-10 强度评分。

注意：

- 分数主要表达 intensity。
- 分数不直接等同于高级或低级。
- 分数用于对比、记忆检索和文案生成。

第一版维度：

- 香气强度
- 酸质强度
- 甜感强度
- 苦感强度
- 醇厚度
- 干净度
- 层次感
- 余韵长度
- 发酵感
- 平衡感

### 6.4 风味信息填写与联想

风味信息按温度段和感知类型记录。

温度段：

- 高温
- 中温
- 低温

感知类型：

- 香气
- 味道

填写过程中，系统根据用户输入自动联想相关风味名词和描述。

示例：

用户输入“柑橘”，系统可联想：

- 柠檬：明亮、尖锐、清爽、高酸。
- 青柠：更青、更锐利、更有清凉感。
- 甜橙：圆润、亲和、甜感明显。
- 血橙：酸甜中带深色水果感。
- 葡萄柚：果皮感、清苦、尾段微苦。
- 蜜柑：柔和、甜润、日常水果感。
- 柚子：清透、带皮香、略有苦韵。

能力要求：

- 用户可以接受一个或多个联想结果。
- 用户可以拒绝联想结果。
- 联想结果要帮助用户发挥想象力，不强制绑定专业风味轮。
- 系统应避免把用户未确认的联想写成确定事实。

### 6.5 文案生成

输入：

- 咖啡豆信息
- 冲煮参数
- 感官维度评分
- 风味信息
- 用户偏好
- 历史文案
- 外部参考内容

输出：

- 小红书标题
- 正文
- 标签
- 封面文案建议
- 可选图片生成提示词

第一版文案风格：

- 克制版：简短、客观、真实。
- 夸张版：风味丰富、情绪强烈、适合种草。
- 锐评版：表达更直接，可好评也可吐槽。

约束：

- 不虚构用户没有提供或确认的信息。
- 不直接复制外部帖子。
- 尽量避免与历史文案重复。
- 输出前进行自检，标出可能夸张或缺乏依据的表述。

### 6.6 记忆归档

每次形成最终文案后，需要存档。

归档内容：

- 咖啡豆信息
- 冲煮参数
- 感官维度评分
- 高温、中温、低温风味记录
- 用户选择的文案版本
- 最终标题、正文、标签
- 是否发布
- 用户满意度
- 风味关键词
- 风格标签

归档用途：

- 后续生成文案时按需召回。
- 对比类似咖啡。
- 避免重复文案。
- 学习用户长期偏好。

## 7. Agent 能力设计

### 7.1 上下文管理

上下文分为三类：

- 当前任务上下文：本次咖啡、参数、感受、文案目标。
- 外部参考上下文：通过 RAG 检索到的相似小红书帖子或其他公开内容。
- 约束上下文：用户偏好、历史表达、禁止虚构、发布前确认等规则。

当前会话中，系统需要持续维护：

- 已收集字段。
- 缺失字段。
- 用户确认过的信息。
- 用户否认或拒绝的信息。
- 当前草稿版本。

### 7.2 规划模块

规划模块负责识别用户意图并拆解任务。

意图类型：

- 记录一次咖啡。
- 生成文案。
- 查询别人评价。
- 分析冲煮参数。
- 扩展风味描述。
- 生成图片。
- 发布笔记。
- 存档记录。

典型流程：

1. 判断用户当前目标。
2. 检查信息是否足够。
3. 不足时追问。
4. 需要外部参考时调用检索工具。
5. 需要风味扩展时调用风味联想工具。
6. 生成候选文案。
7. 审稿与事实检查。
8. 用户确认。
9. 存档。
10. 如发布能力可用，调用发布工具。

### 7.3 记忆模块

短期记忆：

- 当前会话内容。
- 本次填写状态。
- Agent 已追问的问题。
- 用户刚选择或拒绝的风味联想。
- 当前草稿和修改意见。

长期记忆：

- 用户文案风格偏好。
- 用户口味偏好。
- 曾喝过的咖啡。
- 曾喝到的类似咖啡。
- 历史最终文案。
- 每条记录的风味关键词和感官评分。

长期记忆应支持按需加载，类似 skill 的使用方式：每条历史记录带有描述、关键词和适用场景，只有与当前任务相关时才召回。

### 7.4 工具加载模块

候选工具：

- 豆袋图片解析工具。
- 风味联想工具。
- 感官评分结构化工具。
- 小红书类似内容检索工具。
- 历史记录检索工具。
- 文案相似度检测工具。
- 文案生成工具。
- 图片生成工具。
- 发布工具。

工具调用要求：

- 工具输入输出结构化。
- 工具能力可插拔。
- 发布类工具必须在用户确认后调用。
- 无法调用工具时，应给出降级方案。

### 7.5 多 Agent 协作模块

第一版可以先实现为逻辑角色，后续再拆为真实多 Agent。

角色包括：

- 采访 Agent：负责追问和收集信息。
- 感官 Agent：负责风味、香气、味道、温度段分析。
- 冲煮 Agent：负责参数和萃取复盘。
- 检索 Agent：负责查找相似外部内容。
- 文案 Agent：负责生成标题、正文和标签。
- 审稿 Agent：负责检查重复、虚构和风格偏差。
- 记忆 Agent：负责存档和历史召回。
- 发布 Agent：负责发布前确认和调用发布工具。

## 8. 多模态能力

### 8.1 文本与视觉模型

可用模型：

- `gpt-5.5`：日常对话、信息抽取、规划、文案生成、审稿。
- `gpt-image-2`：图片生成。

### 8.2 图片输入

第一版支持上传豆袋图片。

用途：

- 识别咖啡豆基础信息。
- 识别官方风味描述。
- 作为本次记录的附件。

### 8.3 图片输出

支持根据用户描述或咖啡风味生成图片。

用途：

- 小红书配图。
- 封面图。
- 风味视觉化。

## 9. MVP 范围

第一版建议实现：

- 对话模式。
- 结构化模板模式。
- 咖啡豆信息手动填写。
- 豆袋图片解析。
- 冲煮参数记录。
- 0-10 感官维度评分。
- 高温、中温、低温下的香气与味道记录。
- 风味自动联想。
- 三种文案风格生成。
- 历史记录归档。
- 基于历史记录的相似文案召回。
- 发布包生成。
- 图片生成接口预留或基础接入。

第一版暂缓：

- 真实一键发布小红书。
- 完整外部 RAG 检索链路。
- 复杂多 Agent 编排框架。
- 完整前端视觉工作台。
- 精细化账号运营数据分析。

## 10. 待调研事项

### 10.1 小红书一键发布

需要调研：

- 小红书是否提供官方个人创作者笔记发布 API。
- 商家开放平台或创作服务平台是否支持内容发布。
- 调用条件、认证要求、审核机制、风险限制。
- 非官方自动化发布是否违反平台规则或存在账号风控风险。

当前建议：

- 第一版只做发布包生成。
- 真实发布能力抽象为插件接口。
- 只有在确认合规、稳定、可维护后再实现。

### 10.2 小红书内容检索

需要调研：

- 是否存在官方搜索 API。
- 搜索结果是否允许用于个人创作参考。
- 如何保留来源。
- 如何避免复制或过度依赖外部文案。

当前建议：

- 第一版可以先通过手动输入参考内容或通用网页搜索做验证。
- 后续再实现稳定检索工具。

### 10.3 风味联想体系

需要继续设计：

- 基础风味词库。
- 风味层级。
- 同义词与近义词。
- 正面风味与负面风味。
- 不同温度段常见变化描述。

当前原则：

- 参考专业体系，但不被专业体系限制。
- 优先服务创作灵感和用户表达。

## 11. 成功标准

MVP 成功标准：

- 用户可以在 5 分钟内完成一次咖啡记录并得到可发布文案。
- 用户可以从模糊风味词得到更具体的候选表达。
- 系统能存档每次记录，并在后续召回相似记录。
- 生成文案能明显减少重复表达。
- Agent 架构中至少覆盖上下文、规划、短期记忆、长期记忆、工具调用的基础实现。

## 12. 关键风险

- 小红书官方发布能力不明确，真实一键发布可能不可行。
- 外部内容检索存在平台限制、版权和稳定性问题。
- 风味联想过度发挥可能导致文案脱离真实感受。
- 多 Agent 过早拆分可能增加工程复杂度。
- 如果历史记忆设计不当，可能造成上下文膨胀或召回噪声。
