# 咖啡品鉴创作 Agent 项目宪法

## Core Principles

### I. 真实咖啡体验

系统必须保护用户真实的咖啡品鉴体验。文案生成、风味联想、图片提示词和外部参考内容，必须区分用户已确认事实、模型推断、创作联想和第三方观点。Agent 不得把未确认的豆袋信息、品鉴感受或外部帖子内容表述成事实。理由：本产品的价值是帮助用户更好表达，而不是伪造咖啡记录。

### II. 可追踪的 Agent 状态

任何会影响 Agent 输出的功能，都必须让相关状态可追踪，包括当前上下文、短期记忆、长期记忆召回、工具调用、已接受/已拒绝的风味建议和最终归档记录。长期记忆必须结构化保存咖啡记录、风味关键词、感官评分、用户偏好和最终文案。理由：本项目同时是 Agent 学习项目，决策过程必须可检查、可复盘、可调试。

### III. 工具安全与用户确认

外部工具必须通过显式工具适配器调用，并使用结构化输入输出。所有高影响工具，包括 `xiaohongshu-skills` 的发布、评论、点赞、收藏，以及任何公开、破坏性或可能产生成本的动作，必须在执行前获得用户明确确认。API Key 不得写入 Git、日志、截图或生成文档。理由：平台自动化、公开发布和模型凭证都涉及账号、隐私和成本风险。

### IV. 可验证的垂直切片

实现必须按可独立测试的垂直切片推进。每个切片必须包含清晰用户场景、后端行为、必要的前端行为、持久化影响和验证步骤。Agent 功能必须通过自动测试或明确的人工验证覆盖 prompt 约束、工具输出、记忆召回和用户可见流程。理由：代码存在不等于交付完成，用户必须能够信任并学习系统。

### V. 面向学习的克制架构

项目必须优先采用可理解、可讲清楚的架构，而不是过早抽象。Spring Boot、Spring AI、Maven、PostgreSQL、pgvector、React、Vite 和 TypeScript 是默认技术栈，除非有记录在案的技术决策变更。多 Agent 行为应先作为应用内逻辑角色实现，再按真实复杂度拆分为独立运行时。所有新增或修改的方法、函数和构造器必须包含契约注释，明确说明方法作用、每个输入参数的语义，以及返回值或无返回值副作用；Java 默认使用 Javadoc 的方法摘要、`@param` 和非 `void` 方法的 `@return`，异常是调用方契约的一部分时还必须使用 `@throws`。注释必须解释意图和边界，不得只是复述方法名或类型；简单 getter、setter、Java `record` 自动访问器，以及框架或编译器生成的方法可以豁免。理由：本项目既用于交付产品也用于学习 Agent 全链路，稳定的方法契约能降低阅读、评审、调试和后续维护成本。

## 文档语言与知识沉淀

- 项目文档默认必须使用简体中文生成，包括 PRD、架构设计、调研文档、spec-kit 规格、计划、任务、复盘和学习文档。
- 代码标识符、命令、API 字段、英文专有名词、第三方库名和必要引用可以保留英文。
- 面向用户或项目协作的解释性文字必须优先使用中文，除非用户明确要求英文。
- 新文档必须放在 `docs/` 或 spec-kit 约定目录中，不得散落在项目根目录。
- 文档必须说明“已确认事实、来源依据、Codex 判断、待用户确认假设”的边界。
- 技术设计必须同时承担学习材料职责，让用户能理解系统为什么这样设计、正在做什么、下一步该如何学习。

## Product and Technical Constraints

- 产品文档必须按 `docs/README.md` 的目录约定归档。
- 规格必须映射到咖啡品鉴创作 Agent 的真实用户行为：对话、模板记录、风味扩展、记忆、发布或视觉生成。
- 前端工作台必须展示足够的中间状态，让用户理解 Agent 正在做什么。
- PostgreSQL 保存结构化记录；启用向量检索时，pgvector 保存语义记忆。
- `gpt-5.5` 是默认文本模型，`gpt-image2` 是默认图像模型。真实 API Key 只能通过本地环境变量或本地配置提供。
- 小红书搜索和发布应使用本地 `xiaohongshu-skills` 适配器。官方 API 可用性仍属于调研事项，但不取消用户确认要求。

## Development Workflow and Quality Gates

每个功能在实现前必须通过以下关卡：

1. 规格关：定义可独立测试的用户故事、验收场景、数据实体、边界情况和可衡量结果。
2. 计划关：说明功能如何满足所有宪法原则，包括状态追踪、工具安全、中文文档和验证方式。
3. 任务关：按可独立测试的用户故事拆分任务，并覆盖后端、前端、持久化、prompt/工具、记忆、文档和验证任务。
4. 评审关：说明已验证内容、未验证内容，以及剩余的平台、模型或用户体验风险。

任何改变已确认技术栈的决策，都必须先记录到 `docs/architecture/technology-decisions-v0.1.md` 或其后续版本，再让代码依赖该决策。

## Governance

当本宪法与较低层级约定冲突时，以本宪法为准；系统、平台和安全规则除外。任何修订必须说明修改原因、版本变更、受影响原则或章节，并同步更新相关 spec-kit 模板和运行时指南。

版本号遵循语义化规则：

- MAJOR：移除原则，或以不兼容方式重新定义原则。
- MINOR：新增原则、章节，或实质扩展治理规则。
- PATCH：不改变含义的措辞澄清、错别字修复或格式修正。

所有规格、计划、任务列表和实现评审都必须检查是否符合本宪法。违反宪法只允许在明确记录理由、被拒绝的替代方案和后续缓解计划后发生。
