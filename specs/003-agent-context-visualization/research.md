# 调研：Agent 上下文与记忆输入可视化

## 1. 工作台状态边界

**Decision**：扩展现有 `WorkbenchSnapshot`，新增 Agent 状态快照，而不是新增独立 AgentTrace 服务或真实模型链路。

**Rationale**：003 的目标是让用户在当前工作台中理解“哪些内容来自当前会话、哪些是候选记忆、哪些未来会发给模型”。现有工作台已经有刷新快照、对话消息、当前记录和草稿契约，扩展该快照能让页面一次性获得可视化所需状态，并保持本地单用户 MVP 的简单性。

**Alternatives considered**：

- 新增独立 AgentTrace/SSE 服务：更接近未来实时轨迹，但会扩大范围，并把本切片从可视化变成流式架构建设。
- 只做前端静态卡片：实现快，但无法验证后端如何定义来源、发送状态和事实边界。

## 2. 候选记忆来源

**Decision**：候选记忆第一版来自本地示例、当前会话派生或稳定测试数据，并在每条卡片中标明“非真实长期记忆召回”。

**Rationale**：规格明确当前不接长期数据库。候选记忆仍需可见，是为了验证未来召回结果如何进入或不进入模型输入。使用稳定示例可以覆盖相关性、冲突、候选原因和发送状态，同时避免用户误以为 pgvector 或长期记忆已经上线。

**Alternatives considered**：

- 接入真实 PostgreSQL/pgvector：违反当前范围，会引入持久化和检索质量问题。
- 完全不展示候选记忆：无法验证候选记忆与当前上下文、模型输入预览之间的边界。

## 3. 上下文预览发送状态

**Decision**：上下文预览按来源分组，并为每项展示发送状态：`WILL_SEND`、`PAGE_ONLY`、`SEND_AFTER_CONFIRMATION`、`EXCLUDED`。

**Rationale**：用户需要区分“页面上看得到”和“未来会发给大模型”。发送状态比单纯的卡片颜色更明确，也便于测试未确认联想不得作为已确认事实发送。

**Alternatives considered**：

- 只展示一段拼接后的 prompt：接近真实输入，但不利于学习来源边界。
- 只展示是否发送的布尔值：过于粗糙，无法表达待确认后发送和冲突排除。

## 4. 模拟模型输出与事实边界检查

**Decision**：模型输出第一版使用稳定模拟输出或固定样例；事实边界检查按表达片段给出依据类型、风险等级、来源指向和建议动作。

**Rationale**：当前目标是验证审查链路和 UI 边界，不是追求真实文案质量。稳定样例可以确保测试覆盖“有用户依据”“待确认联想”“无依据”“冲突表达”等类别。

**Alternatives considered**：

- 接入真实模型：超出范围，引入凭证、成本、延迟和随机性。
- 只显示审查摘要：不能帮助用户定位具体哪句话越界。

## 5. 新建记录 / 清空当前会话

**Decision**：提供明确的会话控制动作。清空当前会话必须先提示影响范围；确认后重置当前工作台会话、前端本地恢复状态和可见 Agent 状态，不删除长期记忆、历史归档或外部平台数据。

**Rationale**：刷新恢复会保留旧会话和旧草稿，这是 002 的便利性。但 003 需要用户能主动切断当前上下文边界，否则无法判断新输入是否仍受旧上下文影响。

**Alternatives considered**：

- 刷新即清空：不符合 002 的错误恢复和刷新恢复目标，容易丢失用户输入。
- 清空后端全部内存会话：范围太大，且未来多会话或历史记录会有误删风险。

## 6. 页面布局边界

**Decision**：中间主工作区只保留与大模型的对话流、输入框和必要错误提示；当前记录、草稿、Agent 模式和状态卡片都放在右侧“当前记录与 Agent 状态区”。

**Rationale**：用户明确要求彩色卡片放到当前记录下方空白，中间只保留对话框。这个布局也能降低认知负担，让对话和状态观察分工清楚。

**Alternatives considered**：

- 维持四栏：状态可见但过度分裂，并且与用户最新布局意图冲突。
- 把上下文预览放入中间对话流：会让用户难以分辨模型对话与系统状态。
