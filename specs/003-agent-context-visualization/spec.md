# 功能规格：Agent 上下文与记忆输入可视化

**功能分支**：`003-agent-context-visualization`

**创建日期**：2026-06-30

**状态**：Draft

**输入**：用户描述：“Agent 上下文与记忆/模型输入可视化。目标是在真实 Web 工作台的当前记录下方空白区域，以彩色状态卡片展示当前会话上下文、已确认事实、待确认联想、候选记忆、将发送给模型的上下文预览、模型输出和事实边界检查结果；中间主工作区只保留与大模型的对话框。当前不接真实模型、不接长期数据库、不接小红书，只把链路和边界可视化，让我能区分哪些是当前上下文、哪些是候选记忆、哪些未来会发给大模型。后续需要提供“新建记录 / 清空当前会话”入口，让刷新恢复出的旧草稿和旧上下文可以被用户主动清掉。”

**语言要求**：除代码标识符、命令、API 字段和第三方专有名词外，本文档必须使用简体中文。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 分清 Agent 状态来源（优先级：P1）

作为项目作者，我希望在真实 Web 工作台的当前记录下方，以彩色 Agent 状态卡片同时看到当前会话上下文、已确认事实、待确认联想和候选记忆，并且每一类信息都有明确标签、来源说明和是否会进入模型输入的状态；中间主工作区只保留与大模型的对话框，让我能判断 Agent 当前“知道什么”“只是联想到什么”“准备参考什么”。

**为什么是这个优先级**：项目宪法要求可追踪的 Agent 状态。当前记录下方的状态卡片区是用户理解 Agent 链路的主要入口，同时避免中间对话区被状态卡片挤占；如果不能先分清事实、联想和候选记忆，后续模型输入预览、输出检查和长期记忆接入都会缺少可信基础。

**独立测试**：只实现这一项时，用户应能打开工作台，在一次品鉴会话中输入咖啡体验，并在当前记录下方看到各状态卡片随会话变化更新；中间主工作区只展示对话流和输入框；候选记忆必须被明确标记为“候选”或“示例”，不得与已确认事实混在一起。

**验收场景**：

1. **假如** 用户打开工作台且当前没有会话内容，**当** 用户查看当前记录下方的 Agent 状态卡片区，**那么** 卡片区展示空状态，并说明当前没有可用于模型输入的会话上下文或候选记忆。
2. **假如** 用户输入“今天喝了一支水洗埃塞，有柑橘和红茶感”，**当** 状态卡片刷新，**那么** 状态卡片把原始描述列为当前会话上下文，把“水洗埃塞”“柑橘”“红茶感”等用户明确表达的内容列为已确认事实或用户陈述事实。
3. **假如** 系统基于“柑橘”产生“甜橙、青柠、葡萄柚”等扩展，**当** 用户查看状态卡片，**那么** 这些扩展必须显示在待确认联想中，不得出现在已确认事实中。
4. **假如** 状态卡片展示候选记忆，**当** 用户查看每条候选记忆，**那么** 每条都必须显示候选原因、与当前会话的关系、当前是否会进入模型输入，以及它不是长期数据库真实召回的边界说明。

### 用户故事 2 - 预览未来会发给模型的上下文（优先级：P1）

作为项目作者，我希望在发送真实模型之前，能看到一份结构化的上下文预览，清楚知道当前会话上下文、已确认事实、待确认联想和候选记忆中哪些内容未来会发给大模型，哪些只是页面上辅助观察的信息。

**为什么是这个优先级**：本切片的核心目标是把模型输入边界可视化。即使当前不接真实模型，用户也需要先验证“未来会发送什么”和“不会发送什么”，以便后续接入真实模型时减少黑盒风险。

**独立测试**：用户在不配置真实模型凭证的情况下，也能通过工作台查看上下文预览；预览内容必须按来源分组，并能指出每组内容的发送状态。

**验收场景**：

1. **假如** 当前会话已有用户描述、已确认事实和待确认联想，**当** 用户查看上下文预览，**那么** 预览必须按来源分组展示未来模型输入，并对每组标注“会发送”“仅供页面观察”或“待用户确认后发送”。
2. **假如** 某条候选记忆尚未被选入模型输入，**当** 用户查看上下文预览，**那么** 预览必须清楚显示该候选记忆未进入将发送内容。
3. **假如** 待确认联想尚未被用户确认，**当** 用户查看上下文预览，**那么** 预览不得把它们表达为用户已经确认的事实。
4. **假如** 当前切片没有接真实模型，**当** 用户查看上下文预览，**那么** 页面必须明确说明这是未来模型输入预览，不代表已经调用真实模型。

### 用户故事 3 - 查看模拟输出与事实边界检查（优先级：P2）

作为项目作者，我希望在当前不接真实模型的情况下，也能看到模型输出区域和事实边界检查结果，用稳定的模拟输出验证系统如何发现“已确认事实”“待确认联想”“缺少依据的表达”和“需要用户确认的内容”。

**为什么是这个优先级**：事实边界检查是保护真实咖啡记录的关键能力。先用模拟链路展示输出和检查结果，可以让用户验证规则和展示方式，再决定后续如何接入真实模型。

**独立测试**：用户无需真实模型、长期数据库或小红书账号，也能触发一份模拟模型输出，并看到事实边界检查结果指出哪些句子受事实支持、哪些句子超出已确认事实或需要确认。

**验收场景**：

1. **假如** 用户已经形成上下文预览，**当** 用户触发或查看模型输出区域，**那么** 页面展示一份明确标记为模拟链路的模型输出，不得暗示已调用真实模型。
2. **假如** 模拟输出中包含“甜橙爆汁感很明显”但用户只确认了“柑橘感”，**当** 用户查看事实边界检查，**那么** 检查结果必须把“甜橙爆汁感很明显”标为超出已确认事实或待确认表达。
3. **假如** 模拟输出引用了用户明确输入的豆子处理法或风味，**当** 用户查看事实边界检查，**那么** 检查结果应标为有用户陈述依据，并指向对应的上下文来源。
4. **假如** 检查结果发现高风险事实混淆，**当** 用户查看状态卡片区，**那么** 页面必须突出显示风险，并提示该内容在确认前不应进入最终记录。

### 用户故事 4 - 理解切片边界与不可用状态（优先级：P3）

作为项目作者，我希望当前记录下方的状态卡片区能明确说明当前切片没有接真实模型、长期数据库和小红书，因此候选记忆、模型输出和检查结果都是用于验证链路的本地可视化状态，避免我误以为系统已经完成真实召回、真实推理或外部平台动作。

**为什么是这个优先级**：边界说明能降低验收误解，也符合工具安全和学习型项目的要求。该故事不改变主流程价值，但能让用户更放心地推进后续计划。

**独立测试**：用户打开工作台后，无论是否已有会话，都能在当前记录下方的状态卡片或相关分区看到当前能力边界；当外部依赖不可用时，页面以“未接入/未执行”表达，而不是失败或假装成功。

**验收场景**：

1. **假如** 用户查看候选记忆分区，**当** 当前切片尚未接长期数据库，**那么** 页面必须说明候选记忆来自本地示例、当前会话派生或稳定测试数据，而不是真实长期记忆召回。
2. **假如** 用户查看模型输出分区，**当** 当前切片尚未接真实模型，**那么** 页面必须说明输出是模拟结果或固定样例。
3. **假如** 用户查看工具状态，**当** 当前切片尚未接小红书，**那么** 页面必须显示未执行外部发布、评论、点赞、收藏或搜索动作。
4. **假如** 某个分区暂时没有内容，**当** 用户查看状态卡片区，**那么** 页面必须给出可理解的空状态，而不是隐藏分区导致用户无法判断链路是否存在。

### 用户故事 5 - 新建记录并清空当前会话（优先级：P2）

作为项目作者，我希望在工作台中有清楚的“新建记录 / 清空当前会话”入口，当刷新恢复出旧会话、旧草稿或旧上下文时，可以主动开始一条新的咖啡记录，并让中间对话区、当前记录、状态卡片区和模型输入预览都回到空状态。

**为什么是这个优先级**：本功能强调上下文边界可见。用户必须能控制当前会话边界，否则刷新恢复会让旧草稿和旧上下文继续影响当前观察，难以判断哪些内容属于本次记录。

**独立测试**：用户在已有会话和草稿的状态下，点击“新建记录 / 清空当前会话”并确认后，页面应进入新的空记录状态；刷新页面后，不应再次恢复被清空的旧会话。

**验收场景**：

1. **假如** 当前会话已有对话、已确认事实、待确认联想、草稿或状态卡片，**当** 用户触发“新建记录 / 清空当前会话”，**那么** 页面必须提示该动作会清空当前会话可见状态，并要求用户确认。
2. **假如** 用户确认清空当前会话，**当** 清空完成，**那么** 中间对话区回到初始提问，当前记录显示空状态，状态卡片区显示没有可发送上下文，草稿和模型输出不再显示。
3. **假如** 用户取消清空，**当** 回到工作台，**那么** 当前会话、草稿、上下文预览和状态卡片必须保持不变。
4. **假如** 用户清空当前会话后刷新页面，**当** 工作台重新加载，**那么** 页面不得恢复被清空的旧会话或旧草稿。
5. **假如** 当前切片展示候选记忆，**当** 用户清空当前会话，**那么** 清空动作只影响当前会话和本地恢复状态，不得暗示删除真实长期记忆或外部平台数据。

### 边界情况

- 当当前会话为空时，当前记录下方的状态卡片区必须展示空状态，并说明没有内容会进入模型输入预览。
- 当用户在已有内容时触发“新建记录 / 清空当前会话”，系统必须先说明将清空哪些当前会话状态，并允许用户取消。
- 当用户清空当前会话后刷新页面时，系统不得再次恢复已清空会话的草稿、对话消息、当前记录或模型输入预览。
- 当用户只输入风味联想但没有豆子、冲煮或个人确认时，系统必须把这些内容标为用户陈述或待确认，不得自动归档成完整事实。
- 当候选记忆与当前会话事实冲突时，系统必须标出冲突，并默认不把冲突候选写入将发送给模型的上下文。
- 当待确认联想数量较多时，状态卡片区必须保持可扫描，不得让用户无法区分已确认事实和待确认内容。
- 当模拟输出包含缺少依据的咖啡产区、处理法、风味强度、用户偏好或发布建议时，事实边界检查必须把对应表达标为无依据或需确认。
- 当真实模型、长期数据库或小红书能力尚未接入时，页面必须使用“未接入”“模拟”“候选”“预览”等边界词，避免制造真实调用已经发生的错觉。
- 当状态卡片数据刷新失败或暂时不可用时，页面必须保留上一份可见状态并说明其更新时间或不可用原因。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统必须在真实 Web 工作台的当前记录下方提供 Agent 状态卡片区，集中展示当前会话上下文、已确认事实、待确认联想、候选记忆、上下文预览、模型输出和事实边界检查结果。
- **FR-002**：系统必须为状态卡片区中的每一类信息显示明确标签、来源说明、当前状态和是否会进入模型输入的说明。
- **FR-003**：系统必须让中间主工作区只保留与大模型的对话流、输入框和必要错误提示，不得把 Agent 状态卡片、候选记忆卡片或上下文预览卡片放在中间主工作区。
- **FR-004**：系统必须把用户明确输入或确认的咖啡信息与系统生成的待确认联想分开展示，不得把待确认联想列入已确认事实。
- **FR-005**：系统必须展示当前会话上下文，包括用户最近输入、助手追问或回复、当前会话状态，以及这些内容与模型输入预览的关系。
- **FR-006**：系统必须展示已确认事实清单，至少覆盖咖啡豆信息、冲煮信息、用户确认风味、用户偏好或用户明确表达的创作要求中当前可获得的内容。
- **FR-007**：系统必须展示待确认联想清单，并说明这些联想需要用户确认后才可被当作事实使用。
- **FR-008**：系统必须展示候选记忆清单，并说明每条候选记忆的来源边界、候选原因、相关性说明和当前是否进入模型输入预览。
- **FR-009**：系统必须提供将发送给模型的上下文预览，并按来源分组展示哪些内容会发送、哪些不会发送、哪些待确认后才可能发送。
- **FR-010**：系统必须在上下文预览中明确说明当前切片不会调用真实模型，预览只用于验证未来模型输入边界。
- **FR-011**：系统必须展示模型输出区域；在未接真实模型时，该区域必须明确标记为模拟输出或固定样例。
- **FR-012**：系统必须展示事实边界检查结果，区分有用户陈述依据、来自候选记忆、来自待确认联想、无依据和与已确认事实冲突的表达。
- **FR-013**：系统必须在事实边界检查结果中为每条风险表达提供可理解的原因和对应的来源指向或缺失说明。
- **FR-014**：系统必须在候选记忆、模型输出和小红书相关分区明确标示当前未接长期数据库、未接真实模型、未接小红书，并不得展示真实外部动作已执行的状态。
- **FR-015**：系统必须提供空状态、加载状态、刷新失败状态和内容过多时的可扫描展示方式，让用户始终能判断链路是否存在及当前状态是否可信。
- **FR-016**：系统必须避免在页面、日志、验收材料或状态卡片中展示真实 API Key、账号凭证、Cookie、会话令牌或可识别的外部平台隐私数据。
- **FR-017**：系统必须提供可独立验收的样例会话，使用户无需真实模型、长期数据库或小红书账号也能验证完整状态链路。
- **FR-018**：系统必须提供“新建记录 / 清空当前会话”入口，让用户可以主动结束当前会话边界并开始新的空记录。
- **FR-019**：当当前会话已有对话、事实、联想、候选记忆使用状态、草稿、模型输出或事实边界检查结果时，系统必须在清空前提示影响范围并允许用户取消。
- **FR-020**：用户确认清空后，系统必须清除当前会话的可见对话、当前记录、草稿、状态卡片、模型输入预览、模拟输出和事实边界检查结果，并展示新的空状态。
- **FR-021**：用户清空当前会话后，刷新工作台不得恢复已清空的旧会话；清空动作不得被表述为删除真实长期记忆、历史归档记录或外部平台数据。

### 宪法对齐 *(必填)*

- **真实性**：用户输入、用户确认和模拟输出必须分层展示；待确认联想、候选记忆和无依据表达不得被写成已确认咖啡事实。
- **可追踪性**：本功能必须让当前会话上下文、已确认事实、待确认联想、候选记忆、上下文预览、模拟输出和事实边界检查结果在当前记录下方全部可见，并能说明每项内容如何影响或不影响模型输入；中间主工作区必须保持为对话区。
- **工具安全**：本功能不执行公开发布、评论、点赞、收藏、真实搜索、真实生图或真实模型调用；任何未来高影响工具动作都只显示为未接入或待确认，不得自动执行。
- **验证方式**：必须通过自动化验证或人工验收覆盖状态分区、来源标签、模型输入预览、模拟输出标识、事实边界检查、空状态、冲突候选、清空当前会话和未接入外部依赖说明。
- **学习价值**：本功能帮助用户学习 Agent 上下文管理、短期状态、候选记忆、模型输入构造、模型输出审查和事实边界检查之间的关系。
- **中文文档**：本功能相关规格、计划、任务、调研、验收说明和学习材料默认使用简体中文。

### 关键实体 *(涉及数据时填写)*

- **Agent 状态卡片区快照**：表示某一时刻当前记录下方状态卡片区展示的完整状态，包含更新时间、会话状态、分区摘要和边界说明。
- **会话上下文项**：表示当前会话中的用户输入、助手回复、追问、选择或状态变化，包含内容、角色、来源、时间和是否进入模型输入。
- **已确认事实**：表示用户明确提供或确认的咖啡事实，包含事实类型、事实内容、来源上下文和确认状态。
- **待确认联想**：表示系统基于当前事实扩展出的风味、表达或创作方向，包含联想内容、触发依据、确认状态和使用限制。
- **候选记忆**：表示可能与当前会话相关的历史偏好、样例记录或本地测试数据，包含候选原因、来源边界、相关性和是否进入模型输入。
- **上下文预览**：表示未来会发送给大模型的结构化内容预览，包含来源分组、发送状态、排除原因和边界提示。
- **模型输出快照**：表示当前切片中展示的模拟输出或固定样例，包含输出内容、输出来源标识和生成状态说明。
- **事实边界检查结果**：表示对模型输出或候选表达的检查结果，包含表达片段、依据类型、风险等级、来源指向和建议动作。
- **会话控制动作**：表示用户主动新建记录、清空当前会话或取消清空的操作，包含动作类型、影响范围、确认状态和清空后的恢复状态。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：用户打开工作台后，能在 30 秒内找到当前记录下方的 Agent 状态卡片区，并说出至少 4 类状态分区的含义。
- **SC-002**：在验收样例中，用户能在 2 分钟内区分当前会话上下文、已确认事实、待确认联想和候选记忆。
- **SC-003**：在至少 5 条验收样例状态项中，每条都显示来源、状态和是否进入模型输入预览。
- **SC-004**：上下文预览必须清楚标出所有将发送内容和未发送内容；验收样例中未确认联想被当作已确认事实发送的次数为 0。
- **SC-005**：模拟输出区域必须 100% 明确标记为模拟或样例；验收过程中用户不会误判为已调用真实模型。
- **SC-006**：事实边界检查必须识别验收样例中至少 3 类表达：有用户依据、待确认联想、无依据或冲突表达。
- **SC-007**：当候选记忆与已确认事实冲突时，系统默认不把冲突候选纳入将发送上下文，并在状态卡片区中显示冲突原因。
- **SC-008**：在真实模型、长期数据库和小红书未接入的情况下，用户仍能完成一次完整链路验收：输入会话内容、查看状态分区、查看上下文预览、查看模拟输出、查看事实边界检查。
- **SC-009**：用户在已有草稿和上下文的会话中，能在 30 秒内完成“新建记录 / 清空当前会话”并回到空记录状态。
- **SC-010**：清空当前会话后刷新页面，旧会话、旧草稿和旧模型输入预览恢复次数为 0。

## 假设

- 第一阶段目标用户是项目作者本人，本地单用户使用，不新增登录、注册、多用户权限或远程部署要求。
- 本切片基于已有真实 Web 工作台继续推进，重点扩展当前记录下方的 Agent 状态卡片区，不重新定义整套工作台主流程。
- 候选记忆可以来自本地示例、当前会话派生或稳定测试数据；真实长期数据库召回不属于本切片范围。
- 模型输出可以使用模拟结果或固定样例；真实模型调用、模型凭证管理和模型成本控制不属于本切片范围。
- 小红书搜索、发布、评论、点赞、收藏和外部平台数据读取不属于本切片范围。
- “新建记录 / 清空当前会话”只重置当前工作台会话和恢复状态，不删除未来可能存在的长期数据库记录、历史归档或外部平台数据。
- 本切片的价值是验证链路、来源标签和事实边界，而不是追求最终文案质量或真实召回效果。
