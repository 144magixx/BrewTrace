## 用户场景与测试 *(必填)*

### 用户故事 1 - 浏览器完成对话文案切片（优先级：P1）

作为项目作者，我希望打开本地浏览器工作台后，可以创建一次咖啡品鉴会话，输入自然语言咖啡体验，并看到系统追问缺失信息；当我补充豆子、冲煮参数和文案风格后，页面能展示克制版、夸张版和锐评版文案草稿，并明确标出哪些内容是已确认事实、哪些只是联想或待确认候选。

**为什么是这个优先级**：当前项目已经有离线业务内核，但用户不能通过真实页面体验主流程。这个故事把最核心的 Agent 产品价值变成可打开、可交互、可演示的垂直切片，是后续接入记忆、轨迹、数据库和外部工具的基础。

**独立测试**：只实现这一项时，用户应能从空白本地环境启动工作台，在浏览器里完成“创建会话 -> 输入首轮描述 -> 查看追问 -> 补充事实 -> 查看三版草稿”的完整流程，不依赖真实模型、小红书账号或历史数据库。

**验收场景**：

1. **假如** 用户打开本地工作台首页且没有会话，**当** 用户点击或触发开始记录，**那么** 页面创建新会话并展示“今天喝了什么咖啡？”输入入口。
2. **假如** 用户已创建会话，**当** 用户输入“今天喝了一支水洗埃塞，有柑橘和红茶感”，**那么** 页面展示用户消息、助手追问豆子信息、冲煮参数和文案风格，且不生成最终草稿。
3. **假如** 用户已收到追问，**当** 用户补充豆子信息、冲煮参数并要求克制版、夸张版和锐评版，**那么** 页面展示三版文案草稿，并在当前记录区域保留已确认事实和待确认联想的区分。
4. **假如** 草稿包含甜橙、青柠或葡萄柚等扩展词，**当** 用户查看草稿，**那么** 页面必须把这些词标记为联想或待确认候选，不得表述成用户已经喝到的事实。

---

### 用户故事 2 - 用户可见的错误与恢复（优先级：P2）

作为项目作者，我希望当本地服务未启动、请求失败或页面刷新时，工作台能给出清楚提示，并尽量保留我已输入的内容，让我知道下一步该重试、重新创建会话还是检查本地服务。

**为什么是这个优先级**：真实 Web 应用的第一版很容易遇到启动顺序、连接失败和刷新丢状态问题。可恢复体验能显著降低学习和验收成本。

**独立测试**：关闭或中断本地服务后打开页面，用户应看到可理解的错误提示；服务恢复后，用户可以重新尝试创建会话或继续当前可恢复流程。

**验收场景**：

1. **假如** 本地服务不可用，**当** 用户打开工作台或提交消息，**那么** 页面显示服务不可用提示，并保留输入框内容。
2. **假如** 用户已经创建会话，**当** 页面刷新，**那么** 页面应恢复当前会话状态，或明确提示当前会话需要重新创建，不得静默丢失用户输入。
3. **假如** 服务返回分级错误，**当** 前端展示错误，**那么** 页面必须展示错误类别、是否可恢复和建议下一步。

---

### 用户故事 3 - 一键本地验收指引（优先级：P3）

作为项目作者，我希望有清晰的本地验收指引，能按顺序启动后端服务、启动前端页面、执行自动化验证，并知道当前切片已验证和未验证的内容。

**为什么是这个优先级**：这是学习型项目。真实 Web 应用不仅要能跑，还要让用户能复盘每一步发生了什么，避免“能不能跑”变成黑盒。

**独立测试**：用户按照文档从干净工作区执行本地启动和验收步骤，应能在 10 分钟内打开页面并完成用户故事 1 的浏览器验收。

**验收场景**：

1. **假如** 用户从项目根目录开始，**当** 用户按照本地验收指引依次执行启动和测试步骤，**那么** 用户能看到后端服务就绪、前端页面地址和自动化验证结果。
2. **假如** 某个验证步骤失败，**当** 用户查看文档，**那么** 文档应说明常见原因、恢复方式和当前未覆盖的风险。

### 边界情况

- 当本地后端服务未启动或端口不可用时，前端必须显示清楚的连接失败提示，并保留用户当前输入。
- 当用户输入为空、过短或只有无关内容时，系统应提示用户补充咖啡体验，而不是创建无意义草稿。
- 当用户只提供风味但没有豆子或冲煮信息时，系统应继续追问缺失事实，不应直接归档或生成最终记录。
- 当用户刷新页面或重新打开工作台时，系统应恢复当前可用会话状态；如果无法恢复，应明确说明原因和下一步。
- 当文案生成过程中出现联想词时，系统必须继续区分用户确认事实与模型联想。
- 当自动化验证通过但真实模型、数据库、小红书或发布流程尚未接入时，文档必须明确说明未验证范围。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统必须提供一个可在本地浏览器打开的工作台首页，首页默认进入咖啡记录工作流，而不是展示营销页或空白页。
- **FR-002**：用户必须能够从页面创建新的品鉴会话，并看到会话已创建的可见状态。
- **FR-003**：用户必须能够在页面提交自然语言咖啡体验，并在同一页面看到用户消息和助手回复。
- **FR-004**：当用户首轮信息不足时，系统必须追问豆子信息、冲煮参数和文案风格，不得直接把待确认信息写成事实草稿。
- **FR-005**：当用户补充足够事实后，系统必须在页面展示克制版、夸张版和锐评版三类文案草稿。
- **FR-006**：页面必须展示当前记录摘要，至少区分已确认事实、待确认联想和草稿状态。
- **FR-007**：页面必须对连接失败、请求失败和可恢复错误给出用户可理解的提示，并尽量保留用户输入。
- **FR-008**：系统必须提供本地验收指引，说明如何启动、如何验证、哪些能力已覆盖、哪些能力尚未接入真实外部依赖。
- **FR-009**：自动化验证必须覆盖主流程、空输入、信息不足、补充事实生成草稿、错误提示和事实边界展示。
- **FR-010**：本功能不得引入真实公开发布、真实小红书互动或自动生图动作；这些能力只允许在后续独立切片中接入。
- **FR-011**：系统不得把真实 API Key、账号凭证、浏览器 Cookie 或会话令牌展示在页面、日志、测试输出或文档中。

### 宪法对齐 *(必填)*

- **真实性**：用户输入的咖啡体验、豆子信息和冲煮参数是用户确认事实；系统扩展出的甜橙、青柠、葡萄柚等风味词只能作为联想或待确认候选展示；三版文案不得把候选词写成用户已确认事实。
- **可追踪性**：本功能必须让会话状态、用户消息、助手追问、草稿状态、事实边界提示和错误状态可见；若暂不展示完整 Agent 轨迹，也必须保留后续接入轨迹侧边栏的位置和验收说明。
- **工具安全**：本功能默认不执行高影响外部工具；不做公开发布、点赞、评论、收藏或真实生图；若出现需要凭证的配置，只能通过本地私有配置提供，且不得进入 Git、页面或日志。
- **验证方式**：必须通过自动化测试和浏览器人工验收覆盖主流程、事实边界、错误恢复和刷新/重启行为；真实模型、小红书和数据库能力不作为本切片默认验收条件。
- **学习价值**：本功能帮助用户学习如何把离线 Agent 业务内核接成真实可交互产品，重点观察“页面动作 -> 服务命令 -> Agent 编排 -> 用户可见状态”的链路。
- **中文文档**：本功能相关规格、计划、任务、验证记录和学习说明默认使用简体中文。

### 关键实体 *(涉及数据时填写)*

- **工作台会话**：表示用户当前正在进行的一次咖啡记录流程，包含会话标识、状态、对话消息、当前草稿和错误状态。
- **对话消息**：表示用户输入或助手回复，包含角色、内容、来源类型和创建时间。
- **当前记录摘要**：表示页面上可见的咖啡豆、冲煮参数、确认风味、待确认联想和草稿状态。
- **文案草稿**：表示克制版、夸张版和锐评版文案，包含标题、正文、标签、事实边界说明和审稿提示。
- **错误提示**：表示用户可见的失败状态，包含错误类别、是否可恢复、建议下一步和保留的用户输入。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：用户从启动本地服务到打开工作台首页，能在 10 分钟内完成首次浏览器验收。
- **SC-002**：用户在浏览器中输入首轮咖啡体验后，3 秒内看到助手追问或可恢复错误提示。
- **SC-003**：用户补充足够事实后，页面能展示 3 类文案草稿，且每类草稿都包含事实边界说明。
- **SC-004**：在信息不足的场景中，系统直接生成最终草稿的次数为 0。
- **SC-005**：在验收样例中，未确认风味被表述成用户事实的次数为 0。
- **SC-006**：当本地服务不可用时，页面能保留用户输入并展示恢复建议，用户无需重新输入同一段内容。
- **SC-007**：本地自动化验证覆盖至少 6 个用户可见场景：打开首页、创建会话、提交首轮消息、展示追问、补充事实生成草稿、错误提示。

## 假设

- 第一阶段目标用户仍是项目作者本人，本地单用户使用，不新增登录、注册、多用户权限或远程部署。
- 本切片优先打通对话文案主流程，不要求同时接入模板记录、长期记忆、真实轨迹侧边栏、发布包或小红书真实工具。
- 当前离线业务内核的对话、追问、草稿生成和事实边界规则可以作为本切片的业务事实来源。
- 本地验收允许使用稳定假数据或离线模型适配器，不要求真实模型 API Key。
- 页面视觉应延续既有三栏工作台设计，但本规格只要求完成可交互垂直切片，不要求一次性完善所有页面模块。
