## Clarifications

### Session 2026-06-30

- Q: 系统如何判断一支咖啡是否与历史记录重复？ → A: 不做强去重，只提示“可能相似”，由用户手动确认是否重复。
- Q: MVP 的小红书发布边界是什么？ → A: 用户二次确认后，可以自动点击发布。
- Q: 系统能否自动写入用户长期偏好？ → A: 可以自动写入，但用户之后可以编辑或删除。
- Q: MVP 每次外部参考检索最多摘要多少条内容？ → A: 每次最多摘要 5 条，并优先使用封装后的 `xiaohongshu-skills` 能力。
- Q: MVP 的图片能力范围是什么？ → A: 支持豆袋图片解析；仅在用户主动请求生图时，根据用户提示词和已生成文案生成配图候选。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 对话记录并生成文案（优先级：P1）

作为咖啡博主，用户可以用自然语言描述一次咖啡体验。系统通过必要追问补齐关键信息，区分用户确认事实和系统联想，并生成可用于小红书的标题、正文、标签和发布前提示。

**为什么是这个优先级**：这是最直接缓解用户“不想写文案”和“相似风味词穷”痛点的主流程，也是后续记忆、工具和审稿能力的入口。

**独立测试**：只实现该故事时，用户输入一次模糊咖啡体验，系统能够追问、整理事实、生成至少一个可用文案草稿，并标注哪些内容来自用户确认。

**验收场景**：

1. **假如** 用户只描述“今天喝了一支水洗埃塞，有柑橘和红茶感”，**当** 用户请求生成文案，**那么** 系统必须识别信息不足并追问关键字段。
2. **假如** 用户确认豆子信息、冲煮参数和口味感受，**当** 用户选择克制版文案，**那么** 系统必须生成简短、客观、不过度夸张的标题、正文和标签。
3. **假如** 系统提出“甜橙、青柠、葡萄柚”等风味联想，**当** 用户未选择其中某项，**那么** 最终文案不得把该项写成已确认事实。

---

### 用户故事 2 - 结构化模板完成品鉴记录（优先级：P1）

作为咖啡博主，用户可以通过模板记录咖啡豆信息、冲煮参数、感官维度评分和不同温度段的香气/味道表现。填写风味时，系统根据当前输入给出可选择、可拒绝、可编辑的风味联想。

**为什么是这个优先级**：结构化记录是长期记忆、相似咖啡对比和稳定评价体系的基础，能让用户每次记录不失偏颇。

**独立测试**：用户完成一次模板填写后，系统能够保存结构化品鉴记录，并基于已选择风味生成文案草稿。

**验收场景**：

1. **假如** 用户输入“柑橘”，**当** 系统生成风味联想，**那么** 必须给出多个更具体候选，并附带简短描述和适用感知维度。
2. **假如** 用户分别填写高温、中温、低温的香气和味道，**当** 系统保存记录，**那么** 记录必须保留温度段和感知类型。
3. **假如** 用户给酸质、甜感、苦感、醇厚度等维度打分，**当** 系统展示记录，**那么** 必须明确这些分数表达强度而非优劣。

**示例**：

用户输入“柑橘”时，系统可以给出如下候选，供用户选择、拒绝或编辑：

- 柠檬：明亮、尖锐、清爽、高酸。
- 青柠：更青、更锐利、更有清凉感。
- 甜橙：圆润、亲和、甜感明显。
- 血橙：酸甜中带深色水果感。
- 葡萄柚：果皮感、清苦、尾段微苦。
- 蜜柑：柔和、甜润、日常水果感。
- 柚子：清透、带皮香、略有苦韵。

这些候选在用户确认前只能作为风味联想，不得直接写入最终事实记录或发布文案。

---

### 用户故事 3 - 记忆归档与相似记录召回（优先级：P2）

作为咖啡博主，用户希望系统记住历史咖啡、最终文案、风味关键词和个人偏好。新建记录或生成文案时，系统可以召回相似咖啡和相似表达，帮助避免重复文案并进行对比。

**为什么是这个优先级**：记忆能力是本项目学习 Agent 全链路的核心，也直接影响长期使用价值。

**独立测试**：用户完成两次带有相似风味的记录后，系统能够在第二次生成文案前召回第一次记录，并提示相似点和应避免重复的表达。

**验收场景**：

1. **假如** 用户确认最终文案，**当** 用户选择存档，**那么** 系统必须保存咖啡信息、冲煮参数、感官评分、风味关键词、文案版本和用户满意度。
2. **假如** 新记录与历史记录有相同风味关键词，**当** 用户请求生成文案，**那么** 系统必须展示相关历史记录摘要。
3. **假如** 新记录与历史记录在豆名、烘焙商、产区、处理法或批次信息上可能相同，**当** 系统发现相似记录，**那么** 系统只提示“可能相似”，并由用户手动确认是否重复。
4. **假如** 用户多次选择克制版文案，**当** 系统更新偏好，**那么** 可以自动写入长期偏好，但必须保留推断依据，并允许用户之后编辑或删除。

---

### 用户故事 4 - 查看 Agent 真实交互过程（优先级：P2）

作为学习 Agent 开发的用户，用户希望在工作台侧边栏中看到本次 Agent 与大模型、工具和记忆系统的真实交互过程，包括传入模型的 prompt、模型决策、工具调用参数、工具返回结果、记忆召回内容和审稿判断。

**为什么是这个优先级**：本项目兼具实用价值和学习价值。该能力能让用户理解 Agent 为什么追问、为什么调用工具、如何使用记忆、如何做文案审稿，是学习上下文管理、规划、记忆和工具调用的关键入口。

**独立测试**：用户完成一次记录或文案生成后，能够打开侧边栏查看该次流程的关键步骤，并看到真实 prompt、工具调用和模型决策摘要。

**验收场景**：

1. **假如** 用户完成一次对话式文案生成，**当** 用户打开 Agent 交互侧边栏，**那么** 系统必须展示本次请求的关键 prompt、模型输出摘要和最终决策路径。
2. **假如** 系统调用了风味联想、历史召回或小红书检索工具，**当** 用户查看侧边栏，**那么** 系统必须展示工具名称、调用目的、输入摘要、返回状态和可读结果摘要。
3. **假如** 本次流程包含敏感信息或用户隐私内容，**当** 系统展示真实交互过程，**那么** MVP 默认不做脱敏，以便用户学习真实 Agent 交互。
4. **假如** 模型因为事实不足而追问或拒绝写入某个风味，**当** 用户查看侧边栏，**那么** 系统必须说明该决策依据来自哪些上下文或约束。

---

### 用户故事 5 - 外部参考与发布确认（优先级：P3）

作为咖啡博主，用户可以询问“别人怎么评价这支咖啡”，系统检索外部相似内容并总结为参考。用户确认最终发布包后，系统可以进入发布准备流程，但公开发布前必须再次确认。

**为什么是这个优先级**：该故事增强创作灵感和发布效率，但涉及外部平台、登录态和账号风险，因此晚于核心记录与记忆。

**独立测试**：用户请求外部参考时，系统能返回来源明确的摘要；用户请求发布时，系统能展示待发布内容并要求确认，不会自动公开发布。

**验收场景**：

1. **假如** 用户询问“别人是什么感受”，**当** 系统检索外部内容，**那么** 必须标明该内容是外部参考，不得混入用户真实感受，且每次最多摘要 5 条相关内容。
2. **假如** 用户确认发布包，**当** 系统进入发布准备流程，**那么** 必须展示标题、正文、标签、图片和风险提示。
3. **假如** 用户已确认发布包并在发布页预览中完成二次确认，**当** 发布工具可用，**那么** 系统可以执行公开发布动作。
4. **假如** 用户未进行最终确认，**当** 发布工具可用，**那么** 系统也不得执行公开发布动作。

---

### 边界情况

- 当豆袋图片解析结果不完整或疑似错误时，系统必须允许用户修改，并标记识别结果为待确认。
- 当用户填写的风味联想与感官评分明显冲突时，系统必须提示用户复核，而不是自动改写事实。
- 当历史记录召回结果过多时，系统必须只展示最相关摘要，并允许用户展开查看。
- 当外部参考内容不可访问、无结果或触发限制时，系统必须降级为无外部参考创作。
- 当外部参考检索结果超过 5 条时，系统必须筛选最相关的最多 5 条用于摘要，其余结果不得进入本次生成上下文。
- 当发布工具未登录、不可用、限流或需要验证码时，系统必须停止发布流程并提示用户手动处理。
- 当用户在发布页预览后取消发布时，系统必须保留发布包或草稿，不得直接丢弃已填写内容。
- 当用户要求生成图片时，系统必须基于用户提示词、已生成文案和已确认风味生成，不得伪造真实杯测结论。
- 当 Agent 交互侧边栏展示真实 prompt 或工具结果时，MVP 默认不做内容脱敏；该能力仅面向本地单人使用。
- 当某次流程没有调用工具或没有召回记忆时，侧边栏必须明确显示“本步骤未调用工具”或“本步骤未召回记忆”，避免用户误解。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统必须支持用户通过自然语言描述一次咖啡体验，并在关键信息缺失时追问。
- **FR-002**：系统必须支持结构化记录咖啡豆信息、冲煮参数、感官维度评分、温度段风味和最终文案。
- **FR-003**：系统必须在风味填写时根据用户输入生成具体风味候选，并支持用户接受、拒绝或编辑候选。
- **FR-004**：系统必须区分用户确认事实、模型联想、外部参考和待确认信息，并在生成文案时保留该边界。
- **FR-005**：系统必须生成至少三种文案方向：克制版、夸张版、锐评版。
- **FR-006**：系统必须在最终文案确认后支持归档本次品鉴记录。
- **FR-007**：系统必须根据咖啡信息、风味关键词、感官评分或历史文案召回相似记录。
- **FR-008**：系统必须在生成新文案时提示可能重复的历史表达。
- **FR-009**：系统必须支持用户查看、接受或忽略召回的历史记录，并且不得自动判定咖啡记录重复。
- **FR-010**：系统必须在发现可能重复的咖啡记录时提示用户手动确认是否重复。
- **FR-011**：系统必须支持上传豆袋图片并提取候选豆子信息，且所有识别结果必须允许用户修改。
- **FR-012**：系统必须支持用户主动触发生图，并基于用户提示词、已生成文案和已确认风味生成配图候选。
- **FR-013**：系统必须支持检索外部相似内容并生成摘要，且摘要必须标明为外部参考；每次最多摘要 5 条相关内容。
- **FR-014**：系统必须生成发布包，包括标题、正文、标签、图片或图片建议、发布前检查项。
- **FR-015**：系统必须在任何公开发布、评论、点赞、收藏或类似高影响动作前获得用户明确确认。
- **FR-016**：系统必须记录每次工具调用的目的、输入摘要、输出状态和是否需要用户确认。
- **FR-017**：系统必须提供用户可理解的 Agent 执行过程说明，包括追问原因、风味联想来源、记忆召回原因和审稿提示。
- **FR-018**：系统必须支持用户记录文案满意度，以便后续更新偏好。
- **FR-019**：系统必须在模型输出缺乏依据、可能过度夸张或与用户事实冲突时给出审稿提示。
- **FR-020**：系统必须在用户确认发布包并完成发布页预览二次确认后，才允许执行自动公开发布。
- **FR-021**：系统必须在发布取消、发布失败或登录受阻时保留发布内容，并向用户说明当前状态和下一步操作。
- **FR-022**：系统可以根据用户多次选择、评分和修改行为自动写入长期偏好，但必须保存推断依据，并允许用户查看、编辑或删除。
- **FR-023**：系统必须优先通过封装后的 `xiaohongshu-skills` 工具能力完成小红书搜索、详情获取、发布页填写和自动发布，而不是让 Agent 直接执行 skill 流程。
- **FR-024**：系统不得在用户未主动请求生图时自动生成图片。
- **FR-025**：系统必须提供 Agent 交互侧边栏，用于展示本次流程的 prompt、模型输出摘要、模型决策、工具调用、工具返回结果、记忆召回和审稿判断。
- **FR-026**：Agent 交互侧边栏在 MVP 阶段默认展示未脱敏的真实交互内容，用于本地单人学习和调试。
- **FR-027**：Agent 交互侧边栏必须按时间顺序展示关键步骤，并标明每一步属于用户输入、上下文组装、模型调用、工具调用、记忆召回、审稿或发布确认。
- **FR-028**：MVP 前端实现必须遵循 [前端工作台设计 v0.1](../../docs/architecture/frontend-design-v0.1.md) 中已确认的信息架构、视觉风格和关键交互，包括三栏工作台、Codex 式对话入口、当前记录紧凑面板、Agent 轨迹彩色卡片流、风味候选 chips、迷你雷达图、文案 Tabs 和发布包检查页。
- **FR-029**：系统必须支持两种 Agent 编排模式：显式工作流模式和模型自主工具调用模式；用户必须能在前端工作台中切换后续对话轮次使用的模式。
- **FR-030**：无论使用哪种 Agent 编排模式，公开发布、评论、点赞、收藏、生图等高影响或产生成本动作都不得跳过用户确认、工具适配器和轨迹记录。
- **FR-031**：Agent 交互侧边栏必须展示每次流程使用的编排模式；在模型自主工具调用模式下，还必须展示模型选择工具的原因。
- **FR-032**：系统必须使用 Kafka 承载归档后记忆写入、偏好推断、工具结果处理和发布流程推进等后置副作用，并通过 PostgreSQL Outbox 表 + 后端定时 Publisher 保证核心业务事务与事件投递的最终一致性。
- **FR-033**：Kafka 消费者必须支持幂等处理；任何由事件触发的高影响动作都必须重新校验用户确认状态。
- **FR-034**：系统必须采用 REST + SSE 通信形态：REST 负责命令和查询，SSE 负责实时推送 Agent 轨迹、工具调用状态、记忆召回结果和异步事件进度。
- **FR-035**：SSE 事件只能推送状态和轨迹，不得替代用户确认命令；公开发布、评论、点赞、收藏、生图等高影响动作仍必须通过 REST 确认接口提交。
- **FR-036**：系统必须采用分级错误与可恢复降级策略，至少包含 `USER_FIXABLE`、`RETRYABLE`、`DEGRADED`、`FATAL`、`SAFETY_BLOCKED` 五类错误。
- **FR-037**：模型失败、外部参考失败、Embedding 失败、Kafka/Outbox 失败、小红书登录受阻、SSE 断线等情况必须保留用户已填写内容和可恢复状态，并在 Agent 轨迹中记录失败原因。

### 宪法对齐 *(必填)*

- **真实性**：用户确认的咖啡事实、系统风味联想、图片创意和外部参考必须分开呈现；未确认内容不得写成事实。
- **可追踪性**：本功能涉及当前会话上下文、短期记忆、长期归档、风味候选、工具调用记录、历史召回摘要和最终文案归档。
- **工具安全**：豆袋图片解析、外部内容检索、图片生成和发布准备都需要记录工具状态；公开发布或互动类动作必须二次确认，确认后才允许自动执行；小红书能力优先封装成工具适配器。
- **验证方式**：通过用户故事验收、文案事实边界检查、记忆召回检查、工具降级场景和发布确认流程验证。
- **学习价值**：本功能覆盖上下文管理、规划、记忆、工具加载、多 Agent 角色、多模态输入输出和审稿流程。
- **中文文档**：本规格以及后续计划、任务、调研、学习材料默认使用简体中文。

### 关键实体 *(涉及数据时填写)*

- **品鉴会话**：表示一次正在进行的咖啡记录过程，包含当前意图、已确认事实、待确认问题、已拒绝建议和当前草稿。
- **咖啡记录**：表示一次完整品鉴归档，包含豆子信息、冲煮参数、感官评分、温度段风味、最终文案和用户满意度。
- **风味候选**：表示系统根据用户输入生成的可选风味，包含名称、描述、温度段、香气/味道维度和用户选择状态。
- **感官评分**：表示各维度 0-10 的强度记录，分数表达强度而非优劣。
- **文案草稿**：表示某次生成的标题、正文、标签和风格版本，包含事实边界和审稿提示。
- **用户偏好**：表示用户对文案风格、口味倾向、表达禁忌和发布习惯的长期记忆。
- **外部参考摘要**：表示来自外部内容的观点摘要，包含来源信息、检索关键词和适用范围。
- **发布包**：表示待用户确认的发布内容集合，包括标题、正文、标签、图片和发布前检查项。
- **工具调用记录**：表示一次工具调用的目的、输入摘要、输出状态、风险等级和确认状态。
- **Agent 交互轨迹**：表示一次 Agent 流程的可学习记录，包含步骤顺序、prompt 摘要、模型决策、工具调用、工具结果、记忆召回和审稿判断。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：用户能够在 5 分钟内完成一次咖啡记录并获得可发布文案草稿。
- **SC-002**：当用户输入一个模糊风味词时，系统能提供至少 5 个具体候选，并包含简短描述。
- **SC-003**：用户能够清楚区分每篇文案中的已确认事实、系统联想和外部参考内容。
- **SC-004**：系统在生成新文案时能展示至少 1 条相关历史记录或明确说明无相关记录。
- **SC-005**：公开发布或外部互动动作在未获得用户最终确认前的执行率必须为 0；用户完成二次确认后，系统可以自动执行发布。
- **SC-006**：用户能在工作台中看到本次 Agent 流程的关键步骤，包括追问、风味联想、记忆召回、工具调用和审稿结果。
- **SC-007**：最终归档记录必须包含咖啡信息、冲煮参数、感官评分、风味关键词和最终文案。
- **SC-008**：在外部参考不可用时，用户仍能完成本次记录和文案生成。
- **SC-009**：系统发现可能重复记录时，必须由用户确认后才标记为重复或关联记录。
- **SC-010**：发布失败、取消或受阻时，用户必须能继续看到发布内容和失败原因。
- **SC-011**：用户能够查看系统自动写入的长期偏好，并能编辑或删除其中任意一项。
- **SC-012**：每次外部参考摘要最多包含 5 条内容，并且每条都能区分为外部参考而非用户事实。
- **SC-013**：在用户未主动请求生图时，系统不得自动生成图片；用户主动请求后，系统必须能基于用户提示词和已生成文案给出配图候选。
- **SC-014**：用户完成一次 Agent 流程后，能够在侧边栏看到至少 5 类关键轨迹信息：用户输入、上下文组装、模型调用、工具或记忆使用、最终决策。
- **SC-015**：侧边栏必须展示未脱敏的真实交互内容，供用户理解 prompt、工具调用、工具返回和模型决策过程。
- **SC-016**：打开工作台首页时，用户必须看到符合前端设计文档的三栏布局：左侧导航、中间“今天喝了什么咖啡？”对话创作入口与当前记录紧凑面板、右侧 Agent 轨迹卡片栏。
- **SC-017**：用户能够在工作台顶部切换“显式工作流”和“模型自主工具调用”两种 Agent 模式，并在右侧轨迹栏中看到本轮实际使用的模式。
- **SC-018**：用户归档咖啡记录后，核心记录必须先保存成功；记忆向量和偏好候选可通过 Kafka 异步完成，失败时可重试且不得重复写入。
- **SC-019**：用户提交一次对话后，工作台能够通过 SSE 在不刷新页面的情况下追加显示至少 3 类实时轨迹事件，例如上下文组装、模型调用、工具调用或记忆召回。
- **SC-020**：当外部参考、Embedding 或 SSE 出现故障时，用户仍能继续完成咖啡记录和文案生成；系统必须展示降级原因和可执行下一步。
- **SC-021**：MVP 自动化测试必须覆盖至少 1 个领域聚合单元测试、1 个 Agent contract test、1 个 PostgreSQL/pgvector 集成测试、1 个 Kafka Outbox 集成测试和 1 个前端 Playwright 工作台流程。
- **SC-022**：MVP 接口在不提供登录凭证的情况下也能完成本地单用户主流程；数据库中的会话、记录、记忆、偏好和事件必须默认关联到 `local-user`。
- **SC-023**：所有 JSON REST 响应必须使用 `{ requestId, data, error }` envelope；成功时 `error=null`，失败时 `data=null` 且错误对象包含错误分类和可恢复信息。
- **SC-024**：后端实现必须遵循 `docs/architecture/backend-design-v0.1.md`；新增学习文档必须放入 `docs/learn/`，并使用简体中文说明关键技术选型的原因和好处。

## 假设

- 第一阶段目标用户是项目作者本人，不包含多用户权限和团队协作。
- MVP 不实现本系统登录鉴权，固定使用 `local-user`；小红书登录态只属于外部平台自动化工具。
- 移动端 App 不属于 MVP 范围，优先提供桌面 Web 工作台体验。
- 小红书真实公开发布属于高风险动作，默认需要发布前预览和最终确认；用户二次确认后允许自动发布。
- 外部内容仅用于创作灵感，不用于复制或替代用户真实感受。
- 系统可以保存用户确认后的历史记录，用于后续召回和偏好学习。
- 用户提供模型访问能力，但真实密钥不会写入规格、代码或版本库。
- 生图是用户主动触发的可选动作，不是每次咖啡记录或文案生成的必经步骤。
- Agent 交互侧边栏仅面向本地单人使用，MVP 不提供敏感内容脱敏能力。
