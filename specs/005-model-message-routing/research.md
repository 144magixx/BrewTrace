# 调研：模型驱动消息路由

## Decision：使用顶层 `messageType` 作为模型路由字段

**Rationale**：`CONVERSATION` 与 `POST` 是整次模型返回的业务决策，而不是单个文案版本的属性。顶层 `messageType` 可以让后端直接判断进入继续对话还是发布草稿链路，避免三个文案变体出现互相矛盾的 `messageType`。

**Alternatives considered**：

- 在每个 `variant` 内放 `messageType`：会导致三版文案可能返回不同类型，后端还需要额外合并和冲突处理。
- 只靠 `variants` 是否为空判断：不够显式，模型可能返回空数组但没有说明下一步该追问什么。
- 只靠自然语言判断：会让前端或后端重新猜模型意图，违背结构化路由目标。

## Decision：拆分 POST 与 CONVERSATION 两套消息模型，并共享 `talk`

**Rationale**：两条链路的数据含义不同。CONVERSATION 需要追问、待确认项和风险提醒；POST 需要三版文案、事实依据、推断和风险提醒。共享 `talk` 能保证前端聊天框用同一个字段展示助手对用户说的话，而不是从多个业务字段中拼接。

**Alternatives considered**：

- 使用一个包含大量可空字段的通用 DTO：实现快，但业务含义不清，容易让 CONVERSATION 携带草稿或 POST 缺少文案。
- 只返回三版文案，把追问作为某个版本的正文：会污染文案草稿，并使前端无法判断是否该展示发布入口。
- 将 `talk` 命名为 `assistantTalk` 或 `displayText`：更工程化，但用户已经明确倾向 `talk`，且语义足够直接。

## Decision：聊天气泡只展示 `talk`

**Rationale**：聊天框承担人机交流，不应展示完整 JSON、事实依据数组或文案正文。POST 的文案应在草稿区域展示，CONVERSATION 的追问应通过 `talk` 展示。这样用户能清楚区分“Agent 在跟我说话”和“系统生成了可编辑草稿”。

**Alternatives considered**：

- POST 时在聊天气泡内展示三版文案：信息密度过高，且与后续发布确认区职责重叠。
- 由前端根据 `messageType` 拼接聊天文案：会让多个端重复业务文案规则，不利于一致性。
- 使用 `body` 作为聊天内容：POST 中 `body` 是文案正文，不适合作为会话引导。

## Decision：提示词由版本化模板和运行时上下文动态组合

**Rationale**：模型路由高度依赖字段含义、事实边界、路由规则和风格要求。提示词如果散落在运行代码中，会难以复盘和迭代；如果只在主提示词中放文件路径，模型无法读取真实风格内容。因此后端需要加载版本化模板，并把当前上下文、动态约束和风格提示词内容组合为最终模型输入。

**Alternatives considered**：

- 在 Java 或 TypeScript 中硬编码提示词：短期方便，但已经被项目规范明确禁止。
- 在 prompt 中写相对文件链接：便于人阅读，但模型不会自动打开文件。
- 每个风格单独调用一次模型：更灵活，但增加成本、延迟和一致性风险，不适合本切片。

## Decision：POST 可以在用户明确放弃补充时生成克制草稿，但必须暴露信息缺口

**Rationale**：用户可能明确表示“没有更多补充了，直接生成”。此时继续追问会阻断用户目标；允许 POST 更符合工作流，但文案必须保持克制，并在事实边界、风险提醒或 `talk` 中说明缺少哪些信息，不能补写未确认内容。

**Alternatives considered**：

- 信息不全时永远 CONVERSATION：事实最安全，但用户明确要求生成时体验僵硬。
- 信息不全时也生成完整营销文案：转化快，但容易伪造咖啡记录。
- 让前端决定是否强制生成：会让模型和后端事实边界失去统一判断。

## Decision：后续模型驱动能力沿用“结构化意图 + 后端校验 + 业务路由”模式

**Rationale**：当前只实现 CONVERSATION 和 POST，但未来图片生成、记忆写入、工具确认、发布确认都需要类似结构。统一模式可以避免每个功能重新发明自由文本判断和前端分支逻辑。

**Alternatives considered**：

- 每个功能各自定义模型返回格式：短期局部简单，长期会造成契约碎片化。
- 引入独立多 Agent 运行时：当前项目仍处于学习型 MVP，复杂度过高。
- 完全由前端根据自然语言识别意图：不可靠，也不利于审计和测试。
