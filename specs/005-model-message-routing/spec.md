# 功能规格：模型驱动消息路由

**功能分支**：`005-model-message-routing`

**创建日期**：2026-06-30

**状态**：Draft

**输入**：用户描述：“后续都用模型驱动的设计模式。系统根据大模型结构化返回分出两条链路：信息完备时返回 POST，展示不同版本文案并进入发布草稿链路；信息不完备时返回 CONVERSATION，由大模型继续和用户交流，直到信息确认完毕。POST 和 CONVERSATION 应该是两套 DTO，都要有一个 talk 字段，前端对话框只展示 talk。提示词必须由后端动态拼接，不能硬编码。”

**语言要求**：除代码标识符、命令、API 字段和第三方专有名词外，本文档必须使用简体中文。

## 用户场景与测试 *(必填)*

### 用户故事 1 - 信息不足时继续追问（优先级：P1）

作为记录咖啡体验的用户，我希望在输入信息不足时，Agent 不要急着生成文案，而是用自然的对话继续问我缺少的关键信息，这样最终文案不会建立在臆造事实上。

**为什么是这个优先级**：这是保护真实咖啡体验和事实边界的核心能力。如果系统在信息不足时直接生成文案，就会把模型联想误写成用户事实。

**独立测试**：只实现 CONVERSATION 链路时，用户输入一条缺少风味、产区、处理法或明确生成意图的记录，系统能够展示一条只包含引导话术的助手回复，并保持继续对话状态。

**验收场景**：

1. **假如** 用户只输入“今天喝了一杯咖啡”，**当** Agent 处理本轮输入，**那么** 系统必须返回 CONVERSATION 类型消息，并在 `talk` 中提出 1-3 个需要用户补充或确认的问题。
2. **假如** 用户输入的是非咖啡饮品记录，例如“我喝了杯可乐”，**当** Agent 处理本轮输入，**那么** 系统必须返回 CONVERSATION 类型消息或边界提醒，并通过 `talk` 告知当前无法生成真实咖啡品鉴文案。
3. **假如** Agent 返回 CONVERSATION 类型消息，**当** 前端展示聊天记录，**那么** 对话框只展示 `talk` 内容，不展示候选文案、内部判断字段或完整结构化数据。

---

### 用户故事 2 - 信息完备时生成发布草稿（优先级：P1）

作为准备发布小红书咖啡笔记的用户，我希望在信息足够或我明确表示不再补充时，Agent 能生成克制版、夸张版和锐评版三版文案，并提示我可以选择和修改，而不是继续追问。

**为什么是这个优先级**：这是模型驱动路由的另一条主链路，直接决定用户能否从对话进入文案生产和发布准备。

**独立测试**：只实现 POST 链路时，用户提供完整咖啡记录并明确要求生成文案，系统能够返回 POST 类型消息，生成三版风格完整且事实边界清晰的草稿。

**验收场景**：

1. **假如** 用户已经提供风味、产区、处理法、冲煮体验等足够信息，并说“给我生成文案吧”，**当** Agent 处理本轮输入，**那么** 系统必须返回 POST 类型消息，并包含三版不同风格的文案草稿。
2. **假如** 用户说“没有更多补充了，直接生成”，**当** Agent 判断信息仍有缺口，**那么** 系统可以返回 POST 类型消息，但必须在事实边界字段中标记缺失信息，并且不得补写未确认事实。
3. **假如** Agent 返回 POST 类型消息，**当** 前端展示聊天记录，**那么** 对话框只展示 `talk`，文案版本在草稿区域展示，发布相关操作必须基于草稿区域触发。

---

### 用户故事 3 - 使用统一消息模型驱动后续链路（优先级：P2）

作为产品设计者和学习 Agent 开发的用户，我希望后续功能都遵循模型驱动消息路由模式：模型返回结构化意图，系统校验并路由到对应链路，让对话、文案、发布、记忆和工具动作都有清晰边界。

**为什么是这个优先级**：该模式会成为后续 Agent 能力扩展的基础，减少前端和业务流程靠自由文本猜测意图的风险。

**独立测试**：在不接入真实发布动作的情况下，系统能根据模型消息类型稳定区分“继续对话”和“生成草稿”，并在状态区展示模型决策、事实边界和下一步动作。

**验收场景**：

1. **假如** 模型返回的消息类型是 CONVERSATION，**当** 系统更新工作台状态，**那么** 工作台必须保持等待用户补充的状态，不生成发布草稿，不展示一键发送入口。
2. **假如** 模型返回的消息类型是 POST，**当** 系统更新工作台状态，**那么** 工作台必须进入草稿可见状态，并展示三版文案及发布前确认入口。
3. **假如** 模型返回结构缺失 `talk`、消息类型不合法或 POST 草稿不完整，**当** 系统解析结果，**那么** 系统必须阻断对应链路，并展示可恢复错误或重新生成提示。

---

### 边界情况

- 当模型返回 POST 但缺少三版文案中的任意一版时，系统必须拒绝进入发布草稿链路，并提示模型输出格式异常。
- 当模型返回 CONVERSATION 但同时携带可发布文案时，系统必须只使用 `talk` 继续对话，不应把附带文案展示为可发布草稿。
- 当模型返回 POST 但文案中包含未确认的豆庄、海拔、处理法、产区、风味或用户偏好时，系统必须在事实边界检查中标记风险，并阻止这些内容作为用户确认事实使用。
- 当用户明确表示“没有更多补充了”但已确认事实很少时，系统可以生成克制草稿，但必须在 `talk` 和风险提示中说明信息有限。
- 当模型输出与用户确认事实冲突时，系统必须优先保留用户确认事实，并将冲突内容标记为不可直接使用。
- 当真实模型调用失败、超时、限流或鉴权失败时，系统必须保留当前会话和用户输入，并给出可恢复提示，不得自动生成替代文案。
- 当提示词模板缺失、版本不匹配或动态约束无法组合时，系统必须阻断模型调用或展示可恢复错误，不能退回散落硬编码提示词。

## 需求 *(必填)*

### 功能需求

- **FR-001**：系统必须把模型返回结果分为两类消息：`CONVERSATION` 和 `POST`。
- **FR-002**：`CONVERSATION` 消息必须表示当前上下文不完备或仍需用户确认，并必须包含一个非空 `talk` 字段作为展示给用户的引导话术。
- **FR-003**：`POST` 消息必须表示当前上下文足以生成发布草稿，或用户明确要求不再补充并进入生成阶段，并必须包含一个非空 `talk` 字段作为展示给用户的生成说明。
- **FR-004**：前端聊天对话框必须只展示模型消息中的 `talk` 字段，不得把文案正文、内部路由字段、事实依据数组或完整结构化返回直接作为聊天气泡展示。
- **FR-005**：当模型消息类型为 CONVERSATION 时，系统必须保持继续对话状态，不生成草稿，不展示一键发送或发布确认入口。
- **FR-006**：当模型消息类型为 POST 时，系统必须展示克制版、夸张版和锐评版三版文案草稿，并允许用户继续查看、选择或修改草稿。
- **FR-007**：POST 草稿必须刚好覆盖 `RESTRAINED`、`EXAGGERATED`、`SHARP_REVIEW` 三种风格，且不得重复或缺失。
- **FR-008**：每个 POST 草稿必须包含标题、正文、标签、事实使用、模型推断、待确认项和风险提醒。
- **FR-009**：系统必须校验模型消息类型、`talk` 字段和 POST 草稿完整性；不合格结果不得进入下一条业务链路。
- **FR-010**：系统必须把用户确认事实、模型推断、待用户确认联想、候选记忆和排除项保持为可追踪边界，不得把未确认内容写成用户事实。
- **FR-011**：系统必须支持基于当前会话、已确认事实、待确认联想、候选记忆边界、排除项、任务目标和风格要求组合模型提示词。
- **FR-012**：模型提示词必须来自版本化、可维护的模板和当前上下文动态组合，不得依赖散落在运行代码或测试中的成段硬编码提示词。
- **FR-013**：系统必须在模型状态区或上下文预览中展示本次模型决策属于 CONVERSATION 还是 POST，以及该决策对应的用户可见结果。
- **FR-014**：系统必须保证 POST 只进入发布草稿或发布前确认链路，不得自动执行公开发布动作。
- **FR-015**：系统必须保证后续新增的模型驱动能力使用同一模式：模型返回结构化意图，系统校验意图，再进入对应业务链路。

### 宪法对齐 *(必填)*

- **真实性**：用户原始输入和用户确认事实是文案事实来源；模型提出的问题、风味猜测和补全方向只能作为 CONVERSATION 引导或待确认项；POST 文案不得把未确认内容写成用户事实。
- **可追踪性**：本功能涉及当前会话、已确认事实、待确认联想、候选记忆边界、排除项、模型消息类型、`talk`、POST 草稿、事实依据、模型推断、待确认项、风险提醒和模型决策状态。
- **工具安全**：本功能不执行真实小红书发布；POST 只进入草稿和发布前确认阶段。任何后续公开发布、点赞、评论、收藏或可能产生成本的动作仍必须二次确认。
- **验证方式**：必须验证 CONVERSATION 只展示 `talk` 且不生成草稿，POST 生成三版草稿且三种风格完整，提示词约束能阻止未确认事实进入文案，异常模型输出会被阻断。
- **学习价值**：本功能帮助学习模型驱动路由、结构化模型输出、上下文边界、提示词模板化、状态机和人机协作确认流程。
- **中文文档**：本功能相关规格、计划、任务、评审和学习材料默认使用简体中文；代码标识符、API 字段、模型枚举和必要英文术语可保留英文。

### 关键实体 *(涉及数据时填写)*

- **模型消息**：一次大模型结构化返回的顶层结果，包含消息类型、`talk`、生成时间、事实边界摘要和后续链路信息。
- **CONVERSATION 消息**：模型消息的一种，表示需要继续向用户确认信息；关键属性包括 `messageType=CONVERSATION`、`talk`、追问问题、待确认项和风险提醒。
- **POST 消息**：模型消息的一种，表示可以生成发布草稿；关键属性包括 `messageType=POST`、`talk`、三版文案、事实使用、推断、待确认项和风险提醒。
- **文案变体**：POST 消息中的单个风格草稿，风格只能是克制版、夸张版或锐评版，并且必须保留事实边界字段。
- **事实边界项**：记录某个表达来自用户确认事实、模型推断、待确认联想、候选记忆或冲突内容，用于审查文案是否越界。
- **提示词模板**：版本化的模型指令来源，用于描述字段含义、事实边界、路由规则、输出结构和风格要求。

## 成功标准 *(必填)*

### 可衡量结果

- **SC-001**：在 10 条信息不足的咖啡记录样例中，至少 9 条应进入 CONVERSATION 链路，并展示清晰的 `talk` 追问，而不是直接生成可发布草稿。
- **SC-002**：在 10 条信息完备且用户明确要求生成的样例中，至少 9 条应进入 POST 链路，并返回克制版、夸张版和锐评版三版完整草稿。
- **SC-003**：用户在 1 分钟内能够从界面判断当前结果是“继续补充信息”还是“草稿已生成”，并知道下一步该输入补充信息还是查看草稿。
- **SC-004**：100% 的聊天气泡只展示 `talk` 字段，不展示内部结构化字段、完整 JSON、事实依据数组或文案变体正文。
- **SC-005**：100% 的 POST 结果在进入发布草稿链路前完成消息类型、`talk` 非空、三版风格完整和事实边界字段存在性检查。
- **SC-006**：当模型返回格式异常、风格缺失、消息类型冲突或提示词模板缺失时，系统在一次交互内给出可恢复提示，并保留用户当前输入和会话状态。
- **SC-007**：用户查看模型状态区时，能够看到本轮模型决策类型、已发送上下文、模型返回摘要和事实边界风险，不需要阅读代码即可理解 Agent 为什么继续追问或生成草稿。

## 假设

- 第一阶段仅支持 `CONVERSATION` 和 `POST` 两类模型消息；图片生成、记忆写入、工具确认、公开发布等后续动作将沿用同一模式扩展，但不纳入本规格交付范围。
- `talk` 作为统一的用户可见对话字段，第一阶段不再另设 `assistantTalk`、`reply` 或 `displayText`。
- POST 进入的是草稿和发布前确认链路，不代表自动发布到小红书。
- 用户明确表达“给我生成文案吧”“没有更多补充了”“直接生成”等意图时，可以触发 POST 判断，但系统仍必须保留信息缺失和事实边界提醒。
- 风格要求来自项目已有小红书风格研究成果，但最终模型输入必须使用可维护的版本化提示词模板和动态上下文组合。
- 当前真实长期记忆数据库和真实小红书发布能力仍不属于本功能的必需依赖。
